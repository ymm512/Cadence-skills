# 项目代码开发规范示例

> **说明**：这是一个适用于 AI 编码助手的项目级代码开发规范示例。
>
> **目标**：不是提供通用语言风格清单，而是帮助 AI 在具体仓库中生成“贴近现有项目、最小风险、便于团队接手”的代码。
>
> **使用方法**：
> 1. 复制此文件到 `.claude/project-rules/` 目录
> 2. 结合项目技术栈、目录结构、历史代码风格进行定制
> 3. 在 `CLAUDE.md` 或 `AGENTS.md` 中引用该文件

---

**版本**: v1.0
**最后更新**: [YYYY-MM-DD]
**适用范围**: [填写模块或目录，例如 backend/api、frontend/app、services/*]
**目标**: 让 AI 在本项目中新增、修改、重构代码时，优先产出与现有仓库风格一致、风险可控、便于人工维护的实现。

---

## 1. 总体原则

### 1.1 先遵循仓库事实，再遵循通用最佳实践

本项目可能已经演进多年，存在历史代码风格差异。AI 编码时按以下优先级执行：

1. 优先遵循当前模块、当前目录、当前功能附近的既有实现方式。
2. 若局部风格不一致，优先复用同类文件中的主流写法。
3. 仅在新增代码或明确重构任务中引入更规范的写法。
4. 不因一次需求顺手统一全仓历史风格。

### 1.2 默认策略是最小改动

- 优先修改已有类、函数、模块，而不是新建抽象层。
- 非必要不重命名类、方法、接口、路由、SQL id、配置项。
- 非任务要求不移动目录结构、不替换框架、不大面积格式化。
- 改动应沿着真实调用链展开，避免只改一层导致上下游不兼容。

### 1.3 AI 输出必须便于人工维护

- 优先写清晰代码，不炫技。
- 命名贴近业务语义，避免无语义缩写。
- 复杂逻辑使用少量注释解释“为什么这样做”。
- 不引入项目未使用的新框架、新范式、新依赖，除非需求明确要求。

## 3. AI 修改代码时的执行规则

### 3.1 新增功能时

- 先判断功能应该落在哪一层，不要默认写进 controller 或 service。
- 先搜索同类实现作为模板，再开始新增代码。
- 新接口、新 DTO/BO、数据访问方法、配置项命名要与周边保持一致。

### 3.2 修改功能时

- 先沿调用链确认影响范围：入口、服务、数据访问、配置、测试是否联动。
- 默认最小改动，不顺手修 unrelated 问题。
- 只修与本次需求直接相关的历史问题。

### 3.3 重构时

- 必须保持对外接口、函数签名、配置名称、数据库契约的兼容性，除非需求明确允许变更。
- 重构目标应聚焦：提炼重复逻辑、拆长函数、补校验、补日志、补测试。
- 不做全仓风格统一式重构。

## 4. 分层与边界规则

> 按项目实际分层填写。本节要写“允许做什么”和“禁止做什么”，不要只写层级图。

### 4.1 推荐调用链

`[入口层] -> [能力/服务层] -> [组合/领域层] -> [数据访问层] -> [数据库/外部系统]`

### 4.2 各层职责

#### 入口层

- 负责 `[例如 HTTP 路由、参数转换、权限校验、调用服务、返回响应]`
- 不直接操作 `[例如 DAO / SQL / 仓储层]`

#### 服务层

- 负责 `[例如 核心业务流程、状态流转、外部系统协调]`
- 可以调用 `[按项目填写]`
- 不负责 `[例如 HTTP 细节 / 页面展示]`

#### 数据访问层

- 只负责数据访问
- 不承载复杂业务判断

### 4.3 严禁事项

- 入口层直接操作数据库
- 数据访问层返回业务状态码语义
- 为了复用引入与当前项目不一致的新架构层
- 未经评估引入新的统一包装结构

## 5. 命名规范

### 5.1 基础命名

| 类型 | 规则 | 示例 |
|------|------|------|
| 类/组件/模块 | PascalCase 或按项目约定 | `UserController` |
| 方法/函数 | camelCase 或按语言约定 | `getUserById` |
| 变量 | camelCase 或按语言约定 | `userName` |
| 常量 | `UPPER_SNAKE_CASE` | `MAX_RETRY_COUNT` |
| 包/目录 | 小写或按项目约定 | `com.example.order` |

### 5.2 项目专属命名

- 控制器/入口层后缀: `[例如 Controller / Handler / Route]`
- 服务实现后缀: `[例如 ServiceImpl / UseCase / Processor]`
- 数据访问后缀: `[例如 DAO / Mapper / Repository]`
- 数据对象后缀: `[例如 PO / Entity / Record / DO]`
- 传输对象后缀: `[例如 DTO / BO / VO / Req / Resp]`

### 5.3 命名原则

- 优先复用项目现有命名体系，不强行引入另一套风格。
- 若仓库同时存在多套命名，新增代码优先贴近同目录主流模式。
- 不要为了“更规范”重命名一整条调用链。

## 6. 接口与返回值规范

> 本节必须填写项目真实返回结构，不能套通用模板。

### 6.1 返回结构

本项目默认返回结构：

```json
{
  "[字段1]": "[例如 code 或 respCode]",
  "[字段2]": "[例如 message 或 respDesc]",
  "[字段3]": "[例如 data，可选]"
}
```

### 6.2 规则

- 新增接口优先复用已有响应对象或基础响应类。
- 不要在已有 `respCode/respDesc` 项目中强行切成 `code/message/data`。
- 成功码、失败码优先从现有常量类或枚举中获取。
- 与同类接口保持字段结构一致。

### 6.3 入参处理

- 优先复用项目现有入参转换方式。
- 入参校验失败时，记录日志并返回明确失败信息或抛项目既有业务异常。

---

## 7. 异常处理规范

### 7.1 使用项目现有异常体系

- 业务校验失败: `[例如 BusinessException]`
- 系统异常: `[例如 ServiceException / RuntimeException / ErrorCodeException]`
- 外部依赖失败: `[按项目填写返回或抛出策略]`

### 7.2 规则

- 不要捕获异常后什么都不做。
- 不要只打印异常，不返回上下文。
- 返回失败对象时，必须设置完整失败信息。
- 抛异常时，异常描述中必须带业务上下文。

### 7.3 异常信息要求

- 使用团队习惯语言。
- 带关键业务标识，如订单号、用户 ID、店铺 ID、任务 ID、外部流水号等。
- 避免只写“系统异常”“处理失败”这类无定位价值的信息。

---

## 8. 日志规范

### 8.1 日志框架

- 使用 `[按项目填写，例如 slf4j / logback / pino / structlog]`
- 禁止使用 `System.out.println`、`printStackTrace` 或同类临时输出方式

### 8.2 日志级别

| 级别 | 使用场景 |
|------|----------|
| `DEBUG` | 调试细节、详细出入参 |
| `INFO` | 核心流程节点、关键状态变化 |
| `WARN` | 异常但流程可继续 |
| `ERROR` | 流程失败、关键依赖异常 |

### 8.3 日志内容要求

- 至少包含一个关键业务标识。
- 记录“动作 + 对象 + 结果”。
- 异常日志必须带堆栈。
- 大对象日志需谨慎，避免无节制打印完整报文。

---

## 9. 注释规范

### 9.1 原则

- 注释解释“业务原因、边界条件、历史兼容”，不要重复代码表面含义。
- 复杂分支、状态流转、平台差异、临时兼容逻辑需要注释。
- 优先少而精的注释。

### 9.2 不建议的注释

- “给变量赋值”
- “调用某方法”
- 与代码逐行重复的说明

---

## 10. 数据访问与 SQL 规范

> 如果项目不使用 SQL/MyBatis，也要替换成对应 ORM/Repository 规则。

### 10.1 通用要求

- 避免 `SELECT *`
- 接口方法名、SQL id、映射配置要一一对应
- 参数类型、返回类型与代码保持一致

### 10.2 动态 SQL / 查询构造规则

- 优先使用项目既有的安全写法
- 动态排序、动态表名、动态条件必须可控
- 默认禁止把外部原始字符串直接拼进 SQL

### 10.3 命名与结构

- 使用项目已有的 `DAO` / `Mapper` / `Repository` 命名
- 不要为了统一风格修改现有 namespace、文件名、接口名

---

## 11. 事务与一致性规范

- 涉及多表更新、状态变更、写后回查、异步通知时，评估事务边界。
- 若项目已有统一事务写法，新增同类逻辑保持一致。
- 不把外部慢调用无脑包进数据库事务。
- 涉及重试、幂等、MQ、回调时，明确补偿或防重策略。

---

## 12. 项目专属规则

> 本节最重要。必须填写本项目特有约束，而不是保留空白。

示例：

- 所有新增控制器必须沿用项目现有参数解析方式
- 所有返回对象必须继承现有基础响应类
- 所有外部平台状态码映射必须写中文注释
- 所有异步消费逻辑必须记录幂等键
- 所有数据库动态排序字段必须来自白名单

---

## 13. 测试与验证要求

### 13.1 AI 改动后的最低验证要求

- 检查 import、依赖、类型、签名是否正确
- 检查调用链是否闭环
- 检查配置、路由、SQL/ORM 映射是否一致

### 13.2 高风险改动重点自查

- 状态流转
- 金额计算
- 权限校验
- 异步消费
- 外部系统同步
- 动态 SQL
- 数据迁移或批量任务

### 13.3 无法自动测试时

至少说明：

- 改动影响范围
- 关键调用链
- 建议人工验证的场景

---

## 14. 禁止事项清单

- 禁止把通用模板硬套到项目现有返回结构
- 禁止仅凭通用框架经验替换项目既有实现方式
- 禁止一次需求扩展成全仓重构
- 禁止无上下文地新增抽象层
- 禁止吞异常
- 禁止输出无业务标识的空泛日志

---

## 15. 给 AI 的执行清单

1. 先定位功能属于哪个模块和哪一层。
2. 搜索同类实现，优先复用周边模式。
3. 确认返回结构、异常体系、日志风格、数据访问命名是否与周边一致。
4. 仅修改与需求直接相关的调用链代码。
5. 检查接口、服务、数据访问、配置、测试是否闭环。
6. 检查异常分支、日志、状态码、动态 SQL 或查询安全性。
7. 输出结果时说明影响范围与验证建议。

这份规范的核心不是“写出最理想的新代码”，而是“在具体项目里写出最不容易出事故、最容易被团队接手的代码”。
