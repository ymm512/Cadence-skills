# 业务规则
{business_rule}

## 测试用例设计原则

### 1. 完整性 (Completeness)
- 覆盖所有正常场景
- 覆盖所有异常场景
- 覆盖边界条件
- 覆盖业务规则的所有分支

### 2. 可追溯性 (Traceability)
- 每个测试用例对应明确的业务规则
- 测试目标清晰
- 验收标准明确

### 3. 独立性 (Independence)
- 测试用例之间相互独立
- 不依赖执行顺序
- 可以单独运行

### 4. 可重复性 (Repeatability)
- 相同输入产生相同结果
- 测试数据明确
- 前置条件清晰

## 测试用例组成

每个测试用例包含:
1. **测试用例 ID**: 唯一标识 (TC001, TC002...)
2. **优先级**: P0 (核心) / P1 (重要) / P2 (一般)
3. **测试名称**: 简洁描述测试目标
4. **业务规则**: 关联的业务规则 ID
5. **前置条件**: 测试开始前的系统状态
6. **测试步骤**: 清晰的操作步骤
7. **测试数据**: 具体的输入数据
8. **预期结果**: 明确的预期输出
9. **后置条件**: 测试完成后的系统状态 (可选)

## 测试场景分类

### Happy Path (正常流程)
测试业务规则在正常情况下的行为:
- 输入数据符合所有约束
- 系统状态正常
- 无异常发生

**示例**:
```
业务规则: 用户密码必须包含大小��字母、数字和特殊字符
Happy Path: 输入 "Abc123!@" → 验证通过
```

### Boundary Values (边界值)
测试业务规则的边界条件:
- 最小值和最大值
- 空值和 null
- 长度边界
- 数量边界

**示例**:
```
业务规则: 密码长度必须在 8-32 位之间
边界值:
  - 7 位密码 → 验证失败
  - 8 位密码 → 验证通过
  - 32 位密码 → 验证通过
  - 33 位密码 → 验证失败
```

### Exception Cases (异常场景)
测试业务规则的异常处理:
- 非法输入
- 违反约束
- 系统错误
- 超时情况

**示例**:
```
业务规则: 用户邮箱必须唯一
异常场景: 注册已存在的邮箱 → 提示"邮箱已被使用"
```

### Negative Testing (负面测试)
测试系统对错误输入的处理:
- 错误的数据类型
- 特殊字符注入
- 超长字符串
- SQL 注入尝试

**示例**:
```
业务规则: 用户名只能包含字母、数字、下划线
负面测试: 输入 "user<script>" → 验证失败,提示格式错误
```

## 测试数据设计

### 有效数据 (Valid Data)
符合所有约束的正常数据:
```json
{
  "valid_email": "user@example.com",
  "valid_password": "Abc123!@",
  "valid_phone": "13800138000"
}
```

### 无效数据 (Invalid Data)
违反约束的错误数据:
```json
{
  "invalid_email": "not-an-email",
  "weak_password": "123456",
  "invalid_phone": "123"
}
```

### 边界数据 (Boundary Data)
临界值数据:
```json
{
  "min_length_password": "Abc123!@",  // 8 位,最小值
  "max_length_password": "Abc123!@" + "x" * 24,  // 32 位,最大值
  "too_short_password": "Abc12!@",  // 7 位,小于最小值
  "too_long_password": "Abc123!@" + "x" * 25  // 33 位,大于最大值
}
```

## 优先级判断

### P0 - 核心功能测试
- 核心业务规则验证
- 主流程 Happy Path
- 影响系统可用性的场景

### P1 - 重要功能测试
- 重要业务规则验证
- 常见异常场景
- 边界条件测试

### P2 - 一般功能测试
- 次要业务规则
- 罕见异常场景
- 性能和用户体验
