# Design - 技术设计

## Overview

基于需求文档（如存在）、存量分析（如涉及存量代码）和审查报告（如 Design Review 后返回修改），设计完整的技术方案。技术方案包含系统架构、数据模型、API 设计、技术选型、风险评估等内容。注意：Design 阶段只生成技术方案，不生成实现计划（实现计划在 Plan 节点生成）。

**支持带着审查报告重新设计：**
- ✅ 如果存在审查报告，优先读取并针对问题修改设计
- ✅ 如果是初次设计，正常设计流程

**关键职责：**
- ✅ 设计系统架构和数据模型
- ✅ 设计API接口和业务流程
- ✅ 技术选型和风险评估
- ✅ 支持带着审查报告重新设计
- ❌ 不生成实现计划（在Plan节点生成）

## When to Use

### 前置条件
- ✅ 已存在需求文档（来自 Requirement 或用户已有文档）
- ✅ 如果涉及存量代码，已有存量分析报告（来自 Analyze 或用户已有文档）
- ✅ 如果是 Design Review 后返回修改，已有审查报告（可选）

### 触发条件
当：
- 用户说"技术设计..."
- 用户说"技术方案..."
- 用户说"架构设计..."
- 已有需求文档，准备进入设计阶段
- 用户直接提供需求，需要技术方案

### 灵活性说明

**可跳过前置节点：**
- ✅ 如果已有需求文档（如 `{date}_需求文档_{功能名称}_v1.0.md`），可直接进入 Design
- ✅ 如果已有存量分析报告（如涉及存量代码），可直接使用，无需重新执行 Analyze
- ✅ 简单需求可以直接进入 Design，无需 Requirement 阶段

## The Process

### 详细流程

```mermaid
graph TB
    A[开始Design] --> A1{是否存在审查报告?}

    A1 -->|是| B[读取审查报告]
    A1 -->|否| C[读取需求文档]

    B --> B1[理解审查问题]
    B1 --> B2[针对性修改设计]

    C --> C1[理解功能需求]
    C1 --> D{是否涉及存量代码?}

    D -->|是| E[读取存量分析报告]
    D -->|否| F[分析技术约束]

    E --> E1[理解现有架构]
    E1 --> F

    B2 --> G[设计系统架构]
    F --> G

    G --> H[设计数据模型]
    H --> I[设计API接口]
    I --> J[绘制业务序列图]
    J --> K[技术选型论证]
    K --> L[存量改造规划]
    L --> M[风险评估]
    M --> N[生成技术方案]
    N --> O{用户确认?}

    O -->|需要调整| P[修改方案]
    P --> N
    O -->|准确| Q[保存产物]
    Q --> R[进入Design Review]
```

### 步骤说明

1. **读取审查报告**（可选）⭐
   - 如果存在审查报告（来自 Design Review），优先读取
   - 理解审查发现的问题（P0/P1/P2）
   - 理解每个问题的修复建议
   - 标记需要修改的设计部分

2. **读取需求文档** ⭐
   - 读取 Requirement 阶段生成的需求文档（如存在）
   - 或读取用户提供的已有需求文档
   - 理解功能需求、业务规则、验收标准

3. **读取存量分析报告**（可选）
   - 如果涉及存量代码，读取 Analyze 阶段的分析报告（如存在）
   - 或读取用户提供的已有存量分析
   - 理解现有架构、依赖关系、技术债务

4. **分析技术约束**
   - 识别技术栈约束（从 CLAUDE.md 读取）
   - 识别性能要求
   - 识别安全要求
   - 识别兼容性要求

5. **设计系统架构** ⭐
   - 设计系统整体架构（分层、模块划分）
   - 绘制架构图（使用 Mermaid）
   - 说明模块职责和依赖关系
   - 设计模式选择
   - **如果有审查报告，优先解决架构审查问题**

6. **设计数据模型** ⭐
   - 设计表结构（字段、类型、长度）
   - 设计索引（主键、唯一索引、普通索引）
   - 设计约束（外键、检查约束、默认值）
   - 绘制 ER 图（实体关系图）
   - 考虑分库分表策略（如需要）
   - 考虑数据迁移方案（如涉及存量改造）
   - **如果有审查报告，优先解决数据模型审查问题**

7. **设计 API 接口**
   - 定义接口规范（RESTful/GraphQL）
   - 定义请求参数（路径、查询、请求体）
   - 定义响应格式（成功、失败）
   - 定义错误码
   - 考虑版本控制
   - **如果有审查报告，优先解决 API 审查问题**

8. **绘制业务序列图**
   - 使用 Mermaid 绘制核心业务流程的序列图
   - 标注关键交互和时序
   - 识别性能瓶颈

9. **技术选型论证**
   - 选择技术栈和库
   - 说明选择理由
   - 对比备选方案

10. **存量改造规划**（涉及存量代码时）
    - 识别需要改造的模块
    - 设计改造策略（渐进式/一次性）
    - 评估改造风险
    - 设计回滚方案

11. **风险评估**
    - 识别技术风险
    - 评估风险影响
    - 设计应对措施
    - **如果有审查报告，确认风险评估问题已解决**

12. **生成技术方案**
    - 汇总为完整的技术方案文档
    - 不包含实现计划（实现计划在 Plan 节点生成）
    - **如果有审查报告，在文档中标注已解决的问题**

13. **用户确认**
    - 确保技术方案准确可行
    - **如果有审查报告，确认所有 P0 问题已解决**

### 工具使用

**Serena MCP**:

- `write_file` - 保存技术方案

**Mermaid**:
- 绘制架构图
- 绘制 ER 图
- 绘制序列图

## 输入来源

1. **审查报告**：来自 Design Review 阶段（可选：Design Review 后返回修改时）
2. **需求文档**：来自 Requirement 阶段或用户已有文档（可选但推荐）
3. **存量分析报告**：来自 Analyze 阶段或用户已有文档（涉及存量代码时）
4. **CLAUDE.md**：获取技术栈、架构信息（必须）
5. **用户对话**：用户补充技术约束和需求细节

## 动态时间预估

| 复杂度 | 时间范围 | 说明 |
|-------|---------|------|
| 🟢 简单 | 15-30分钟 | 单层架构，直接实现，无存量依赖 |
| 🟡 中等 | 30-60分钟 | 多层架构，需要详细设计，涉及少量存量代码 |
| 🔴 复杂 | 60-120分钟 | 分布式架构，系统交互复杂，涉及大量存量代码 |

## 关键检查清单 ✅

- [ ] 需求理解：是否已读取并理解需求文档（如存在）？
- [ ] 架构设计：是否包含系统架构图和模块划分？
- [ ] 数据模型：是否设计了表结构、索引、约束、ER 图？
- [ ] API 设计：是否定义了接口规范（请求/响应/错误码）？
- [ ] 流程设计：是否绘制了核心业务序列图？
- [ ] 存量改造：是否规划了存量代码的改造方案？（涉及存量代码时）
- [ ] 技术选型：是否说明了技术选型的理由？
- [ ] 风险评估：是否标注了技术风险和应对措施？

## Red Flags ⚠️

| 错误做法 | 正确做法 |
|---------|---------|
| ❌ 没有需求文档就做复杂设计 | ✅ 简单需求可直接设计，复杂需求应有需求文档 |
| ❌ 涉及存量代码但未分析影响 | ✅ 必须先完成存量分析或使用已有分析报告 |
| ❌ 技术方案过于笼统 | ✅ 需要足够详细才能指导开发 |
| ❌ 在 Design 阶段生成实现计划 | ✅ 实现计划应在 Plan 阶段生成 |
| ❌ 数据模型只做概念设计 | ✅ 数据模型必须包含物理设计（表结构、索引、约束等） |
| ❌ 忽略 CLAUDE.md 中的技术栈约束 | ✅ 必须遵循项目技术栈规范 |

## Integration

### 前置依赖
- **requirement**（可选）：提供需求文档，但不是必须，已有需求文档或简单需求可直接进入
- **analyze**（可选）：涉及存量代码时，提供存量分析报告，可用已有报告
- **design-review**（可选）：Design Review 后返回修改时，提供审查报告

### 下一步
- **design-review**：对技术方案进行系统性审查

### 替代方案
- 如果已有技术方案，可直接进入 Design Review
- 简单功能可跳过 Design Review（需用户确认）

### 需要的输入
- 审查报告（来自 Design Review，可选：Design Review 后返回修改时）
- 需求文档（来自 Requirement 或用户已有文档，推荐但非必须）
- 存量分析报告（来自 Analyze 或用户已有文档，涉及存量代码时）
- CLAUDE.md（必须：技术栈约束）

### 与 Design Review 的交互

**初次设计流程：**
```
Design（初次） → Design Review → 发现问题 → 返回 Design（带着审查报告）
```

**带着审查报告重新设计：**
1. **读取审查报告**
   - 优先读取审查报告，理解 P0/P1/P2 问题
   - 标记需要修改的设计部分

2. **针对性修改设计**
   - 架构审查问题 → 修改系统架构设计
   - 数据模型审查问题 → 修改表结构、索引、约束
   - API 审查问题 → 修改接口设计
   - 安全审查问题 → 增加安全措施

3. **生成修改后的技术方案**
   - 在文档中标注已解决的问题
   - 确保所有 P0 问题已解决

4. **重新进入 Design Review**
   - 审查修改后的技术方案
   - 确认问题已解决

**设计优先级（有审查报告时）：**
```yaml
P0 问题（必须解决）:
  - 架构审查问题
  - 数据模型审查问题
  - API 审查问题
  - 安全审查问题

P1 问题（建议解决）:
  - 性能审查问题
  - 可维护性问题
  - 风险评估问题

P2 问题（可选优化）:
  - 兼容性问题
```

## 确认机制

生成技术方案后：
展示技术方案摘要（3-5 个关键决策）
展示数据模型设计（表结构、ER 图）
展示存量改造计划（涉及存量代码时）
展示风险评估
**如果有审查报告，展示已解决的 P0 问题清单**

询问："这个技术方案可行吗？有什么要调整的？"
├── ✅ 可行 → 保存产物，进入 design-review
├── ⚠️ 需要调整 → 修改方案
└── ❌ 不可行 → 重新设计

## 跳过条件

- 已存在完整的技术方案文档
- 极简单功能（如配置修改、简单查询）
- 纯原型开发（探索阶段）
- 用户明确表示不需要

## 与 Requirement 的边界

**Requirement 阶段负责：**
- ✅ 功能清单和用户故事
- ✅ 业务流程和业务规则
- ✅ 验收标准（清晰到可推导测试用例）
- ✅ 存量复用规划（识别可复用模块）

**Design 阶段负责：**
- ✅ 系统架构设计
- ✅ 数据模型设计（物理设计：表结构、索引、约束、ER 图）
- ✅ API 设计（接口规范）
- ✅ 技术选型论证
- ✅ 存量改造方案（详细的改造策略）
- ✅ 风险评估

**关键区别：**
- Requirement 关注"做什么"（业务视角）
- Design 关注"怎么做"（技术视角）
- Design 的数据模型是物理设计，不是概念模型
- Design 不生成实现计划（实现计划在 Plan 阶段）

## 与 Plan 的边界

**Design 阶段负责：**
- ✅ 技术方案（架构、数据模型、API、技术选型）

**Plan 阶段负责：**
- ✅ 实现计划（任务分解、依赖关系、优先级排序、时间估计）

**关键区别：**
- Design 输出：技术方案（1 个文档）
- Plan 输出：实现计划（1 个文档）
- Design 关注技术选型和架构设计
- Plan 关注任务分解和执行计划
