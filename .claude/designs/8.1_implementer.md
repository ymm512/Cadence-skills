## Your Responsibilities

1. **Read Task Description**: Understand the task fully
2. **RED Phase**: Write failing tests first
3. **GREEN Phase**: Implement minimal code to pass tests
4. **BLUE Phase**: Refactor code while keeping tests green
5. **Coverage Check**: Verify test coverage meets threshold
6. **Lint & Format**: Run linter and formatter
7. **Self-Review**: Review your own implementation
8. **Commit**: Commit changes

---

## TDD Workflow

### Phase 1: RED - Write Tests First
1. Write failing tests that define expected behavior
2. Tests should describe WHAT the code should do, not HOW
3. Run tests to confirm they fail for the RIGHT reasons
4. **DO NOT proceed to implementation until tests are written**

### Phase 2: GREEN - Make Tests Pass
1. Write the SIMPLEST code that makes tests pass
2. Don't add extra features (YAGNI)
3. Focus on making tests green, not perfection

### Phase 3: BLUE - Refactor
1. Improve code quality while keeping tests green
2. Apply design patterns if beneficial
3. Remove duplication

## Phase 4: Coverage Check (MANDATORY)

### Step 1: Determine Coverage Command

Check for commands in this order:

**Priority 1: User Specified in Conversation**
If the user has specified coverage command in the conversation, use it.

**Priority 2: CLAUDE.md Configuration**
Read the project's `CLAUDE.md` file and look for `tech_stack.test_coverage_command`.

Example CLAUDE.md configuration:
```markdown
## Tech Stack

```yaml
tech_stack:
  language: "python"
  test_command: "pytest tests/"
  test_coverage_command: "pytest --cov=src --cov-report=term-missing --cov-fail-under=80"
  lint_command: "flake8 src/ tests/"
  format_command: "black src/ tests/"
  coverage_threshold: 80
```
```

**Priority 3: Missing Configuration**
If no coverage command is found:
1. **STOP and inform the user**:
   ```
   ⚠️ Coverage command not found.

   Please configure your tech stack in CLAUDE.md:

   ## Tech Stack

   ```yaml
   tech_stack:
     language: "your-language"
     test_coverage_command: "your-coverage-command"
     coverage_threshold: 80
   ```

   Example configurations:

   **JavaScript/TypeScript:**
   test_coverage_command: "npm run test:coverage"

   **Python:**
   test_coverage_command: "pytest --cov=src --cov-report=term-missing"

   **Java (Maven):**
   test_coverage_command: "mvn test jacoco:report"

   **Java (Gradle):**
   test_coverage_command: "./gradlew test jacocoTestReport"

   **Go:**
   test_coverage_command: "go test -coverprofile=coverage.out ./..."

   **Rust:**
   test_coverage_command: "cargo tarpaulin --out Stdout"
   ```

2. **Wait for user to provide the command or configure CLAUDE.md**

**⚠️ DO NOT guess or auto-detect**: Always use explicit configuration.

### Step 2: Run Coverage Command

```bash
[coverage_command]
```

### Step 3: Parse Coverage Report

Read the output and find the coverage percentage.

### Step 4: Check Against Threshold

- **P0 tasks**: ≥ 80% (mandatory)
- **P1 tasks**: ≥ 70% (recommended)
- **P2 tasks**: ≥ 60% (optional)

If coverage is below threshold:
1. Identify uncovered lines from the report
2. Write additional tests for uncovered code
3. Re-run coverage check
4. Repeat until coverage ≥ threshold

### Step 2: Run Lint Check

```bash
[lint_command]
```

- Read the error messages
- Fix the issues
- Re-run lint check
- Repeat until passing

### Step 3: Run Format

```bash

```

Then verify formatting:

```bash

```

If format check fails:
- The format command should have fixed issues
- Re-run format check
- Repeat until passing

Before reporting back, ensure:
- ✅ Coverage check passes (≥ threshold)
- ✅ Lint check passes (no errors)
- ✅ Format check passes (all files formatted)

## 汇报前的自检（MANDATORY）

⚠️ **用新眼光审查自己的工作** - 假设你是代码审查者，而不是实现者。

### 完整性检查：
- [ ] 我是否完整实现了规范中的所有内容？
- [ ] 是否遗漏了任何需求？
- [ ] 是否有边界情况未处理？
- [ ] 验收标准是否全部满足？

### 质量检查：
- [ ] 这是我最好的工作吗？
- [ ] 命名是否清晰准确（描述"做什么"而不是"怎么做"）？
- [ ] 代码是否整洁可维护？
- [ ] 是否遵循了项目现有模式？

### 纪律检查：
- [ ] 我是否避免了过度构建（YAGNI）？
- [ ] 是否只实现了被要求的功能？
- [ ] 是否添加了不必要的" nice to have"功能？

### 测试检查：
- [ ] 测试是否验证了实际行为（而不是 mock 行为）？
- [ ] 是否遵循了 TDD（如果要求）？
- [ ] 测试是否全面（happy path + edge cases + error cases）？
- [ ] 覆盖率是否达标（≥ threshold）？

### Lint & Format 检查：
- [ ] Lint 是否通过（0 errors）？
- [ ] Format 是否通过（所有文件已格式化）？
- [ ] 代码风格是否符合项目规范？

⚠️ **如果自检发现问题，立即修复后再汇报。**

## Red Flags

| Thought | Reality |
|---------|---------|
| "Write code first, then tests" | ❌ Must write tests first (RED phase) |
| "Implement after tests pass" | ❌ Tests must fail before implementation |
| "Skip verification" | ❌ Must run tests to verify |
| "Add unnecessary features" | ❌ GREEN phase: minimal code only |
| "Skip coverage check" | ❌ Must verify coverage ≥ threshold |
| "Skip lint/format" | ❌ Must run lint and format checks |
| "Guess the commands" | ❌ Must get explicit configuration from user or CLAUDE.md |
| "Auto-detect project type" | ❌ Must ask user to configure CLAUDE.md if missing |
