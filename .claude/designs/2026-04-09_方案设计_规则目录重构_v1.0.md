# 规则目录重构设计文档

> 日期：2026-04-09 | 版本：v1.0 | 状态：设计中

## 1. 背景与目标

### 现状问题

1. `CLAUDE.md` 约 620 行，7 大类规则全部内嵌，文件过大难以维护
2. 框架内置规则和用户自定义规则混在同一文件中，界限不清
3. `project-rules/` 目录定位不够明确，缺乏强制约束
4. 涉及规则创建的 commands/skills 缺乏统一的规则存储规范

### 目标

1. 新增 `.claude/rules/` 目录，存放框架内置规则
2. CLAUDE.md 只保留摘要 + 引用，详细规则拆分到 `rules/` 下的独立文件
3. 增加强制规则：用户自定义规则只能存放在 `project-rules/` 目录
4. 全面审查 commands/skills，确保规则创建遵循新规范

## 2. 目录结构设计

### 2.1 rules 目录（框架内置规则）

```
.claude/
├── rules/                          # 框架内置规则（维护者管理）
│   ├── README.md                   # 规则目录说明
│   ├── language.md                 # 语言规则
│   ├── code-usage.md               # 代码使用规则
│   ├── document-storage.md         # 文档存储规则（含命名规范、路径映射、禁止行为）
│   ├── markdown-format.md          # Markdown 格式规则
│   ├── serena-usage.md             # Serena MCP 使用规则
│   ├── mcp-servers.md              # MCP Server 使用规则（所有 MCP 工具）
│   └── playwright.md               # Playwright CLI 使用规则
```

### 2.2 project-rules 目录（用户自定义规则）

```
.claude/
├── project-rules/                  # 用户自定义规则（用户管理）
│   ├── README.md                   # 用户规则说明（含强制约束）
│   └── examples/                   # 规则模板示例
│       ├── coding-standards.md
│       ├── design-template.md
│       ├── requirement-template.md
│       └── test-standards.md
```

### 2.3 两个目录的职责边界

| 维度 | `rules/` | `project-rules/` |
|------|----------|-------------------|
| 管理者 | 框架维护者 | 用户 |
| 内容 | 通用规则（语言、文档、MCP 等） | 项目个性化规则（编码规范、模板等） |
| 修改权限 | 仅维护者 | 用户自由修改 |
| 创建方式 | 安装脚本自动创建 | 用户按需创建或从 examples 复制 |

## 3. CLAUDE.md 重构设计

重构后的 CLAUDE.md 精简为 ~80-100 行，结构如下：

```markdown
# CLAUDE.md

本文件为 Claude Code (claude.ai/code) 在此仓库中工作提供指导。

## 强制规则

> 详细规则见 `.claude/rules/` 目录下的各规则文件。
> 用户自定义规则见 `.claude/project-rules/` 目录。

### 1. 语言规则
- **必须使用中文回答** → 详见 `.claude/rules/language.md`

### 2. 代码使用规则
- **非必要不编写代码**（Skills 项目特殊规定） → 详见 `.claude/rules/code-usage.md`

### 3. 文档存储规则
- **所有文档必须存放在 `.claude` 目录下** → 详见 `.claude/rules/document-storage.md`

### 4. Markdown 格式规则
- **代码块嵌套使用 4 反引号/3 反引号** → 详见 `.claude/rules/markdown-format.md`

### 5. Serena 使用规则
- **禁止分析 .git 目录** → 详见 `.claude/rules/serena-usage.md`

### 6. MCP Server 使用规则
- **各 MCP 工具的使用规范** → 详见 `.claude/rules/mcp-servers.md`

### 7. 项目个性化规则（强制规则）
- **用户自定义规则只能存放在 `.claude/project-rules/` 目录**
- 禁止在 `rules/` 目录中添加用户自定义规则
- 禁止直接修改 `rules/` 目录下的框架内置规则文件
- 详见 `.claude/project-rules/README.md`

### 8. Playwright CLI 使用规则
- **浏览器自动化工具规范** → 详见 `.claude/rules/playwright.md`

## 项目信息
# currentDate
Today's date is {当前日期}。
```

### 关键设计决策

1. 每个规则只保留 **1 行摘要 + 文件引用路径**
2. 新增 **第 7 条强制规则**，明确 `project-rules/` 是用户自定义规则的唯一合法位置
3. 明确禁止用户修改 `rules/` 目录

## 5. Commands/Skills 修改方案

### 5.1 必须修改的文件

#### `cadence-init/commands/rule-config.md`

**当前行为**：直接将完整规则文本写入 CLAUDE.md

**修改为**：

1. 创建 `.claude/rules/` 目录
2. 将各类规则内容写入对应的 `rules/*.md` 文件
3. CLAUDE.md 只写入摘要 + 引用（按第 3 节设计的结构）
4. 步骤 6（目录结构创建）增加 `rules/` 目录

#### `cadence-init/commands/project-rules-examples.md`

**当前行为**：创建 `project-rules/` 示例文件

**修改为**：

1. 在步骤 6（添加 CLAUDE.md 规则）中更新引用方式
2. 增加"强制规则"说明：用户自定义规则只能放 `project-rules/`
3. 引用 `references/project-rules/CLAUDE-RULE.md` 更新为摘要 + 引用格式

#### `cadence-init/references/project-rules/CLAUDE-RULE.md`

**当前行为**：包含完整的个性化规则模板

**修改为**：

1. 保留个性化规则的使用说明
2. 增加强制约束：
   - 用户自定义规则只能存放在 `.claude/project-rules/` 目录
   - 禁止在 `.claude/rules/` 目录中添加用户自定义规则
   - 禁止直接修改 `.claude/rules/` 目录下的框架内置规则文件

#### `cadence-init/references/project-rules/README.md`

**当前行为**：`project-rules/` 目录说明

**修改为**：

1. 增加强制约束说明
2. 明确 `rules/` 和 `project-rules/` 的职责边界
3. 增加"禁止行为"章节

### 5.2 需要审查的文件

以下文件需要检查是否有硬编码的规则路径引用或规则创建逻辑：

| 文件 | 审查重点 |
|------|---------|
| `cadence-init/commands/pre-check.md` | 是否检查规则路径 |
| `cadence-init/commands/project-analysis.md` | 是否引用规则路径 |
| `cadence-init/commands/mcp-configuration.md` | 是否写入 MCP 规则到 CLAUDE.md |
| `cadence-init/commands/skill-create.md` | skill 创建是否涉及规则路径 |
| `cadence-workflow/commands/*.md` | 是否有硬编码规则路径 |

### 5.3 审查标准

对所有 commands/skills 的审查遵循以下标准：

1. **规则存储**：框架内置规则 → `rules/`，用户自定义 → `project-rules/`
2. **CLAUDE.md 引用**：只写摘要 + 引用，不写规则原文
3. **强制约束**：明确告知用户 `project-rules/` 是唯一合法的自定义规则目录
