## Skill Definition

```yaml
---
name: using-cadence
description: Use when starting any conversation - establishes how to find and use cadence skills, requiring Skill tool invocation before ANY response
---
```

<EXTREMELY-IMPORTANT>
If you think there is even a 1% chance a cadence skill might apply to what you are doing, you ABSOLUTELY MUST invoke the skill.

IF A SKILL APPLIES TO YOUR TASK, YOU DO NOT HAVE A CHOICE. YOU MUST USE IT.

This is not negotiable. This is not optional. You cannot rationalize your way out of this.
</EXTREMELY-IMPORTANT>

## 核心作用

**using-cadence 是 Cadence Skills 系统的"守门员"和"路由器"**，确保 Claude 在处理开发任务时：

1. ✅ **强制检查** - 在响应之前检查是否有相关 Skill
2. ✅ **智能路由** - 将用户意图映射到正确的 Skill
3. ✅ **流程保护** - 防止跳跃开发流程的关键步骤
4. ✅ **灵活调用** - 提供双通道调用机制

## Skill Priority

When multiple skills could apply, use this order:

### 优先级规则

**P1 - 理解现状类 Skills（优先执行）**
- cadence-analyze - 理解现有代码/架构
- cadence-brainstorm - 探索需求和方案
- cadence-requirement - 明确业务需求

**P2 - 规划设计类 Skills（第二优先）**
- cadence-design - 技术方案设计
- cadence-design-review - 设计审查
- cadence-plan - 实现计划
- cadence-using-git-worktrees - 环境隔离

**P3 - 执行实现类 Skills（最后执行）**
- cadence-subagent-development - 开发实现
- cadence-test-driven-development - TDD
- cadence-test-design - 测试设计
- cadence-integration - 集成测试
- cadence-requesting-code-review - 代码审查
- cadence-deliver - 交付部署
- cadence-verification-before-completion - 完成验证
- cadence-troubleshoot - 问题排查

### 冲突解决规则

**规则 1：理解 > 设计 > 实现**
```
用户说："重构这段代码，优化性能"
→ P1: cadence-analyze（分析现状）
→ P2: cadence-design（设计方案）
→ P3: cadence-subagent-development（实现）
```

**规则 2：存量代码优先分析**
```
用户说："存量代码的需求不明确"
→ 优先：cadence-analyze（先理解存量代码）
→ 然后：cadence-brainstorm（再探索新需求）
```

**规则 3：设计先于实现**
```
用户说："直接帮我实现这个功能"
→ 检查：是否有设计文档？
→ 没有：cadence-design（先设计）
→ 有：cadence-subagent-development（再实现）
```

## 与 Subagent 的关系

### 架构层次

```
using-cadence（元 Skill）
    ↓ 路由到
Cadence Skills（知识层）
    ↓ 可能调用
Subagents（执行层）
```

### 关系说明

**using-cadence（元 Skill）**：
- **作用**：管理 Cadence Skills 的使用
- **职责**：
  - 检查是否有相关 Skill
  - 路由用户意图到对应 Skill
  - 防止绕过 Skills 系统
- **不负责**：具体实现、审查、测试

**Cadence Skills（知识层）**：
- **作用**：定义开发流程各阶段的知识和方法
- **包含**：brainstorm, analyze, design, plan, implement, review, test, deliver 等
- **职责**：
  - 提供 Prompt 模板
  - 定义工作流程
  - 指导如何执行

**Subagents（执行层）**：
- **作用**：独立的 AI Agent，执行具体任务
- **包含**（定义在 8.1/8.2/8.3）：
  - Implementer Subagent - 实现代码
  - Spec Reviewer Subagent - 规格审查
  - Code Quality Reviewer Subagent - 代码质量审查
- **职责**：
  - 独立上下文执行任务
  - 两阶段审查（spec + quality）
  - 自主提问和实现

### 调用流程示例

```
用户："帮我实现用户认证功能"
    ↓
using-cadence（检查 + 路由）
    ↓
cadence-brainstorm（探索需求）
    ↓
cadence-design（设计技术方案）
    ↓
cadence-plan（制定实现计划）
    ↓
cadence-subagent-development（Skill）
    ↓ 调用
Implementer Subagent（8.1）- 实现代码
    ↓ 调用
Spec Reviewer Subagent（8.2）- 规格审查
    ↓ 调用
Code Quality Reviewer Subagent（8.3）- 质量审查
```

## Red Flags

These thoughts mean STOP—you're rationalizing:

| Thought | Reality |
|---------|---------|
| "This is just a simple question" | Questions are tasks. Check for skills. |
| "I need more context first" | Skill check comes BEFORE clarifying questions. |
| "Let me explore the codebase first" | Skills tell you HOW to explore. Check first. |
| "I can check git/files quickly" | Files lack conversation context. Check for skills. |
| "Let me gather information first" | Skills tell you HOW to gather information. |
| "This doesn't need a formal skill" | If a skill exists, use it. |
| "I remember this skill" | Skills evolve. Read current version. |
| "This doesn't count as a task" | Action = task. Check for skills. |
| "The skill is overkill" | Simple things become complex. Use it. |
| "I'll just do this one thing first" | Check BEFORE doing anything. |
| "This feels productive" | Undisciplined action wastes time. Skills prevent this. |
| "I know what that means" | Knowing the concept ≠ using the skill. Invoke it. |

## Skills 不适用场景

以下场景**不需要**使用 Skills：

### 简单操作
- 文件读取（Read file）
- 格式转换（JSON → YAML）
- 简单查询（"这个文件有多少行"）

### 非开发任务
- 翻译文本
- 解释概念（非开发相关）
- 聊天闲谈

### 判断标准

```yaml
✅ 使用 Skills：
  - 涉及开发流程（需求、设计、实现、测试、部署）
  - 需要结构化方法
  - 涉及代码变更
  - 需要审查或验证

❌ 不使用 Skills：
  - 单纯的文件操作
  - 信息查询
  - 非技术任务
  - 已完成流程的小修补
```

## Example Workflow

### 示例 1：新功能开发

```
用户："我想做个用户认证功能"

using-cadence：
  识别关键词"想做个"
  → 路由到 cadence-brainstorm

cadence-brainstorm：
  探索需求
  → 生成需求文档

using-cadence：
  检测到需求已明确
  → 路由到 cadence-design

cadence-design：
  设计技术方案
  → 生成设计文档

using-cadence：
  检测到设计完成
  → 路由到 cadence-plan

cadence-plan：
  制定实现计划
  → 生成任务列表

using-cadence：
  检测到计划完成
  → 路由到 cadence-subagent-development

cadence-subagent-development：
  调用 Implementer Subagent（8.1）实现
  调用 Spec Reviewer Subagent（8.2）审查
  调用 Code Quality Reviewer（8.3）检查
```

### 示例 2：存量代码重构

```
用户："重构这段代码，优化性能"

using-cadence：
  识别关键词"重构"、"优化性能"
  → 优先级规则：P1（理解现状）
  → 路由到 cadence-analyze

cadence-analyze：
  分析现状
  → 生成分析报告

using-cadence：
  检测到分析完成
  → 优先级规则：P2（规划设计）
  → 路由到 cadence-design

...（后续流程同示例 1）
```
