# Cadence 产物目录迁移概要需求

## 背景

当前 Cadence commands 与 skills 生成的 PRD、设计、计划、报告等产物主要写入项目 `.claude/` 目录。该目录同时承载 Claude Code 的规则、命令和技能配置，导致 Cadence 运行产物与 Claude Code 配置边界不清晰。

本次调整将 Cadence 维护的输出产物迁移到项目根目录下的 `cadence/`，保留 Claude Code 相关配置在 `.claude/`，并在 `rule-config` command 中提供旧项目迁移能力，便于在多个项目中统一验证和迁移。

## 目标

- Cadence 新生成的文档、计划、报告、原型、模型、日志和用户项目规则写入 `cadence/` 目录。
- `rule-config` command 创建新的 `cadence/` 目录结构，并提供历史产物迁移能力。
- `rule-config` command 询问用户是否将 `cadence/` 加入 `.gitignore`，默认不忽略。
- `.claude/rules`、`.claude/commands`、`.claude/skills` 保持 Claude Code 专属用途，不迁移。
- `skill-create` 和 `skill-creator` 创建的 project skill 继续写入 `.claude/skills/<skill-name>/SKILL.md`。

## 迁移范围

以下 Cadence 产物目录从 `.claude/` 迁移到 `cadence/`：

| 旧路径 | 新路径 | 用途 |
| --- | --- | --- |
| `.claude/prds` | `cadence/prds` | 概要需求、brainstorming 输出 |
| `.claude/analysis-docs` | `cadence/analysis-docs` | 分析报告 |
| `.claude/analysis` | `cadence/analysis` | 旧版分析报告目录 |
| `.claude/docs` | `cadence/docs` | 需求、详细文档、技术储备文档 |
| `.claude/designs` | `cadence/designs` | 技术方案、实现计划等现有输出 |
| `.claude/designs-reviews` | `cadence/designs-reviews` | 设计评审 |
| `.claude/plans` | `cadence/plans` | 计划文档、进度追踪文件 |
| `.claude/readmes` | `cadence/readmes` | 开发相关 README 文档 |
| `.claude/modaos` | `cadence/modaos` | 原型截图、设计稿 |
| `.claude/models` | `cadence/models` | 数据模型 |
| `.claude/architecture` | `cadence/architecture` | 架构文档 |
| `.claude/notes` | `cadence/notes` | 开发笔记 |
| `.claude/logs` | `cadence/logs` | 开发日志 |
| `.claude/reports` | `cadence/reports` | 进度报告、开发报告 |
| `.claude/project-rules` | `cadence/project-rules` | 用户自定义规则、模板和规范 |
| `.claude/cache` | `cadence/cache` | Cadence 分析缓存 |

以下目录不迁移：

| 路径 | 保留原因 |
| --- | --- |
| `.claude/rules` | Claude Code 框架内置规则 |
| `.claude/commands` | Claude Code 项目级命令 |
| `.claude/skills` | Claude Code 项目级技能 |

## 行为设计

### 新产物路径

所有 Cadence workflow commands、skills、README 示例和初始化模板中，涉及 PRD、analysis、docs、designs、plans、reports、project-rules 等产物路径的说明迁移到 `cadence/`。

迁移后 Cadence 后续只读写 `cadence/` 新路径，不在 workflow skills 中实现 `.claude/` 回退读取。

### `rule-config` 目录创建

`rule-config` 继续创建 `.claude/rules` 等 Claude Code 必需目录，同时创建以下 Cadence 产物目录：

```bash
cadence/{prds,analysis,analysis-docs,docs,designs,designs-reviews,plans,readmes,modaos,models,architecture,notes,logs,reports,project-rules/examples,cache}
```

`rule-config` 不能把 `.claude/rules`、`.claude/commands`、`.claude/skills` 迁移到 `cadence/`。

### `rule-config` 历史迁移能力

`rule-config` 检测旧路径中是否存在可迁移目录。若存在，向用户展示迁移清单，并在用户确认后将旧目录内容迁移到对应 `cadence/` 目录。

迁移策略：

- 只处理迁移范围表中的目录。
- 目标目录不存在时直接移动旧目录到新目录。
- 目标目录已存在时，保留目标目录现有内容，并提示用户处理同名冲突。
- 迁移完成后报告成功、跳过和冲突项。
- 不删除 `.claude/rules`、`.claude/commands`、`.claude/skills`。
- 不迁移 `skill-create` 和 `skill-creator` 生成的 project skills。

### `.gitignore` 决策

`rule-config` 创建或迁移 `cadence/` 时询问用户是否将 `cadence/` 写入 `.gitignore`。默认选择为不忽略，因为 PRD、设计、计划、用户项目规则等产物通常需要团队协作和版本管理。

若用户选择忽略，则在 `.gitignore` 中添加：

```gitignore
cadence/
```

若用户选择不忽略，则不修改 `.gitignore`。

## 影响文件类型

实现时重点检查以下范围：

- `cadence-init/commands/rule-config.md`
- `cadence-init/references/rules/document-storage.md`
- `cadence-init/references/project-rules/` 中涉及 `.claude/project-rules` 的说明
- `cadence-workflow/commands/` 中展示输出路径的命令文档
- `cadence-workflow/skills/` 中读写 Cadence 产物路径的技能文档
- `readmes/skills/` 中面向用户的路径示例
- `README.md`、`AGENTS.md`、`CLAUDE.md` 中与 Cadence 产物路径相关的说明

实现时不直接修改当前仓库 `.claude/rules/` 下的框架内置规则文件。

## 测试与验证

完成实现后至少执行以下验证：

- 使用 `rg` 确认 Cadence 产物路径不再指向旧 `.claude/` 输出目录。
- 使用 `rg` 确认 `.claude/rules`、`.claude/commands`、`.claude/skills`、`~/.claude/skills` 等保留路径未被误迁移。
- 检查 `skill-create` 和 `skill-creator` 仍保留 `.claude/skills` 输出说明。
- 检查 `rule-config` 文档包含 `cadence/` 创建、历史迁移、`.gitignore` 用户确认且默认不忽略。
- 检查 Markdown 格式和嵌套代码块格式符合项目规则。

## 非目标

- 不实现运行时旧路径回退读取。
- 不把 `.claude/rules`、`.claude/commands`、`.claude/skills` 迁移到 `cadence/`。
- 不改变 project skill 和 global skill 的安装目标。
- 不直接修改当前仓库 `.claude/rules/` 下的框架内置规则文件。
