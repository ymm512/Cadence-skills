## 异常场景定义

异常场景测试验证系统在以下情况下的行为:
- 用户输入无效数据
- 业务规则被违反
- 系统资源不可用
- 外部依赖失败
- 并发和时序问题

---

## 异常场景分类

### 1. 输入验证异常
```yaml
invalid_format:
  description: "输入格式不正确"
  examples:
    - "邮箱格式错误: 'not-an-email'"
    - "手机号格式错误: '123'"
    - "日期格式错误: '2026/02/30'"

missing_required:
  description: "必填字段缺失"
  examples:
    - "未填写邮箱"
    - "未填写密码"
    - "未选择必选项"

length_violation:
  description: "长度超出限制"
  examples:
    - "密码太短: '123'"
    - "用户名太长: 'a' * 100"
    - "描述超过 500 字"

type_mismatch:
  description: "数据类型错误"
  examples:
    - "数量输入文本: 'abc'"
    - "金额输入负数: -100"
    - "日期输入未来日期"
```

### 2. 业务规则违反
```yaml
constraint_violation:
  description: "违反业务约束"
  examples:
    - "余额不足进行支付"
    - "库存不足无法下单"
    - "超过每日限额"

state_violation:
  description: "状态转换非法"
  examples:
    - "已取消订单无法支付"
    - "已审批流程无法撤回"
    - "已发货订单无法修改"

permission_violation:
  description: "权限不足"
  examples:
    - "普通用户访问管理功能"
    - "非本人查看他人信息"
    - "已过期 Token 访问 API"
```

### 3. 系统异常
```yaml
resource_unavailable:
  description: "资源不可用"
  examples:
    - "数据库连接失败"
    - "文件存储服务故障"
    - "缓存服务不可用"

external_failure:
  description: "外部依赖失败"
  examples:
    - "第三方支付接口超时"
    - "短信发送失败"
    - "邮件服务不可用"

timeout:
  description: "操作超时"
  examples:
    - "API 请求超时"
    - "文件上传超时"
    - "数据库查询超时"
```

### 4. 并发异常
```yaml
race_condition:
  description: "竞态条件"
  examples:
    - "同时修改同一数据"
    - "双击提交创建重复数据"
    - "并发扣减库存"

deadlock:
  description: "死锁情况"
  examples:
    - "循环依赖资源锁定"
    - "事务相互等待"
```

## 设计步骤

### Step 1: 识别异常点

分析工作流中的每个步骤:
```yaml
for_each_step:
  input_validation:
    - "哪些输入可能无效?"
    - "哪些字段可能缺失?"

  business_rules:
    - "哪些业务规则可能被违反?"
    - "哪些状态转换是非法的?"

  system_dependencies:
    - "哪些外部服务可能失败?"
    - "哪些资源可能不可用?"
```

### Step 2: 确定错误响应

```yaml
error_response:
  user_facing:
    - "错误提示文案"
    - "错误状态展示"
    - "恢复操作引导"

  system_level:
    - "HTTP 状态码"
    - "错误码"
    - "错误日志"
```

### Step 3: 验证错误处理

```yaml
error_handling:
  graceful_degradation:
    - "系统是否正常处理错误?"
    - "是否有备选方案?"

  data_integrity:
    - "错误是否导致数据损坏?"
    - "事务是否正确回滚?"

  user_experience:
    - "错误信息是否清晰?"
    - "用户是否知道如何修复?"
```

```json

  {
    "test_case_id": "EX-001",
    "name": "登录 - 邮箱不存在",

    "expected_error": {"code": "USER_NOT_FOUND", "status": 401}
  },
  {
    "test_case_id": "EX-002",
    "name": "登录 - 密码错误",
    "exception_category": "business_rule",
    "input": {"email": "user@example.com", "password": "wrong"},
    "expected_error": {"code": "INVALID_PASSWORD", "status": 401}
  },
  {
    "test_case_id": "EX-003",
    "name": "登录 - 账户被锁定",
    "exception_category": "business_rule",
    "precondition": "账户因多次失败已锁定",

  },
  {
    "test_case_id": "EX-004",
    "name": "登录 - Token 过期",

]
```

## 优先级判断

### P0 - 核心异常
- 影响核心业务流程
- 可能导致数据丢失
- 安全相关异常

### P1 - 重要异常
- 影响用户体验
- 常见的错误场景
- 业务规则验证

### P2 - 一般异常
- 罕见的错误场景
- 边缘情况
- 系统级异常
