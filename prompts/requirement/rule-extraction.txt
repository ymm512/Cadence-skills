## 规则分类标准

### 1. 验证规则 (Validation Rules)
**特征**: 输入格式、必填字段、数据约束
**关键词**: "必须", "应该", "格式", "长度", "范围", "不能为空"
**示例**:
- 用户邮箱必须符合标准邮箱格式
- 密码长度必须在 8-32 位之间
- 手机号必须是 11 位数字

### 2. 业务逻辑规则 (Business Logic Rules)
**特征**: 计算公式、条件判断、状态转换
**关键词**: "计算", "如果...则", "当...时", "基于", "根据"
**示例**:
- 订单总价 = 商品单价 × 数量 × 折扣
- 如果用户余额不足,则订单状态变更为"待支付"
- 当订单完成后,自动发放积分

### 3. 权限规则 (Permission Rules)
**特征**: 角色权限、数据访问控制、操作授权
**关键词**: "只有", "仅限", "管理员", "权限", "可以", "不能"
**示例**:
- 只有管理员可以删除用户
- 用户只能查看自己的订单
- 超级管理员拥有所有权限

### 4. 流程规则 (Process Rules)
**特征**: 审批流程、状态机、工作流
**关键词**: "审批", "提交", "流转", "状态", "步骤"
**示例**:
- 订单提交后需要���理员审核
- 审核通过后状态变更为"进行中"
- 取消订单需要提供取消原因

## 提取方法

### 识别模式
1. 寻找包含约束性语言的句子
2. 识别条件判断结构 (if-then, when-then)
3. 定位数学公式和计算逻辑
4. 提取状态转换描述

### 优先级判断
- **P0 (核心)**: 包含"必须"、"关键"、"核心"的规则
- **P1 (重要)**: 包含"应该"、"重要"的规则
- **P2 (一般)**: 包含"可以"、"建议"的规则

### 验证类型
- **format**: 格式验证 (邮箱、手机号、URL)
- **range**: 范围验证 (长度、数值范围)
- **required**: 必填验证
- **unique**: 唯一性验证
- **conditional**: 条件验证 (依赖其他字段)

## 输出格式
每条规则应包含:
- **id**: 规则唯一标识 (BR001, BR002...)
- **description**: 规则描述 (清晰、准确)
- **type**: 规则类型 (validation/business_logic/permission/process)
- **priority**: 优先级 (P0/P1/P2)
- **validation_type**: 验证类型 (如果适用)
- **affected_fields**: 涉及的字段 (如果适用)
- **error_message**: 违反规则时的错误提示

## 示例输出
```json
{
  "id": "BR001",
  "description": "用户邮箱必须符合标准邮箱格式",
  "type": "validation",
  "priority": "P0",
  "validation_type": "format",
  "affected_fields": ["email"],
  "error_message": "邮箱格式不正确"
}
```
