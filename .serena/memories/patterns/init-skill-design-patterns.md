# Init Skill 设计模式

**创建日期**: 2026-02-28
**适用范围**: Cadence 项目初始化

---

## 核心设计模式

### 模式1：智能检测 + 人工确认

**应用场景**：
- 技术栈检测（语言、测试命令、Lint/Format 命令）
- 项目类型检测（前端/后端/全栈）

**模式流程**：
```
1. 自动检测项目特征
   ├── 检测项目文件（package.json、requirements.txt 等）
   ├── 检测测试命令
   ├── 检测 Lint 命令
   └── 检测 Format 命令

2. 生成检测结果
   └── 整理为配置模板

3. 用户确认
   ├── ✅ 确认 → 写入配置
   ├── ✏️ 修改 → 修改后重新确认
   └── ❌ 取消 → 跳过配置

4. 写入配置
   └── 保存到 CLAUDE.md
```

**模式价值**：
- ✅ 自动化：减少手动配置工作
- ✅ 准确性：用户确认确保配置正确
- ✅ 灵活性：支持手动修改

**注意事项**：
- ⚠️ 检测结果必须用户确认
- ⚠️ 提供手动配置选项
- ⚠️ 记录检测逻辑供用户参考

---

### 模式2：跨平台适配

**应用场景**：
- 路径映射（文档存储路径）
- 配置文件位置（MCP 配置）
- 命令格式（包管理器命令）

**模式流程**：
```
1. 检测操作系统
   └── 使用系统 API 或环境变量

2. 选择适配策略
   ├── macOS → 使用 macOS 路径和命令
   ├── Linux → 使用 Linux 路径和命令
   └── Windows → 使用 Windows 路径和命令

3. 生成配置
   └── 根据操作系统生成对应配置

4. 验证配置
   └── 确保路径和命令在目标系统有效
```

**路径映射示例**：
```markdown
| 系统 | 完整路径示例 |
|------|-------------|
| **macOS** | `/Users/{username}/projects/myproject/.claude/docs/` |
| **Linux** | `/home/{username}/projects/myproject/.claude/docs/` |
| **Windows** | `C:\Users\{username}\projects\myproject\.claude\docs\` |
```

**模式价值**：
- ✅ 兼容性：支持多操作系统
- ✅ 自动化：自动适配，无需手动选择
- ✅ 可维护性：集中管理跨平台差异

**注意事项**：
- ⚠️ 测试所有支持的操作系统
- ⚠️ 处理路径分隔符差异（`/` vs `\`）
- ⚠️ 处理命令格式差异（`./gradlew` vs `gradlew.bat`）

### 模式3：渐进式初始化

**应用场景**：
- 整个初始化流程
- 每个步骤独立可恢复

**模式流程**：
```
1. 检测已完成的步骤
   └── 使用状态文件或记忆系统

2. 跳过已完成步骤
   └── 幂等性设计

3. 执行下一步
   └── 独立执行，不依赖前序步骤状态

4. 记录完成状态
   └── 保存到 Checkpoint

5. 继续下一步
   └── 循环直到所有步骤完成
```

**步骤示例**：
```
步骤1: 调用 /cadencing 命令
步骤2: 添加强制规则
步骤3: 检测项目类型
步骤4: 添加包管理器规则
步骤5: 检测技术栈
步骤6: 用户确认技术栈
步骤7: 写入技术栈配置
步骤8: 添加 MCP 配置
步骤9: 创建目录结构
步骤10: 创建 Checkpoint
```

**模式价值**：
- ✅ 可恢复：中断后可继续
- ✅ 幂等性：重复执行安全
- ✅ 可跳过：支持跳过特定步骤

**注意事项**：
- ⚠️ 每个步骤必须独立
- ⚠️ 必须记录完成状态
- ⚠️ 必须支持断点续传

---

### 模式4：用户确认门控

**应用场景**：
- 关键配置写入
- 文件覆盖操作
- 项目类型确认

**模式流程**：
```
1. 展示即将执行的操作
   └── 详细说明操作内容和影响

2. 询问用户确认
   ├── ✅ 确认 → 执行操作
   ├── ✏️ 修改 → 修改后重新询问
   └── ❌ 取消 → 跳过操作

3. 执行操作或跳过
   └── 根据用户选择执行
```

**确认示例**：
```
技术栈检测结果：
- 语言：{language}
- 测试命令：{test_command}
- 覆盖率命令：{test_coverage_command}
- Lint 命令：{lint_command}
- Format 命令：{format_command}
- 覆盖率阈值：80%

是否确认以上配置？
├── ✅ 确认 → 生成配置并写入 CLAUDE.md
├── ✏️ 修改 → 询问需要修改的项
└── ❌ 取消 → 跳过技术栈配置
```

**模式价值**：
- ✅ 安全性：防止误操作
- ✅ 透明性：用户了解操作内容
- ✅ 灵活性：支持修改和取消

**注意事项**：
- ⚠️ 必须展示详细的操作内容
- ⚠️ 必须提供修改和取消选项
- ⚠️ 不能自动跳过用户确认

## 设计模式组合应用

### 场景1：技术栈配置

**使用的模式**：
1. **智能检测 + 人工确认** - 检测技术栈
2. **跨平台适配** - 适配不同操作系统
3. **用户确认门控** - 确认配置写入

**组合流程**：
```
智能检测技术栈（模式1）
    ↓
跨平台适配命令（模式2）
    ↓
用户确认配置（模式3）
    ↓
写入 CLAUDE.md
```

---

### 场景2：目录结构创建

**使用的模式**：
1. **跨平台适配** - 适配路径分隔符
2. **渐进式初始化** - 逐个创建目录

**组合流程**：
```
检测操作系统（模式2）
    ↓
确定路径格式（模式2）
    ↓
逐个创建目录（模式3）
    ↓
记录完成状态（模式3）
```

---

### 场景3：CLAUDE.md 配置更新

**使用的模式**：
1. **用户确认门控** - 确认是否覆盖
2. **渐进式初始化** - 分步添加规则

**组合流程**：
```
检测 CLAUDE.md 是否存在
    ↓
询问用户是否覆盖（模式4）
    ↓
分步添加规则（模式3）
    ├── 添加语言规则
    ├── 添加文档存储规则
    ├── 添加命名规范
    └── 添加其他规则
    ↓
记录完成状态（模式3）
```

## 反模式警示

### 反模式1：自动写入无确认

❌ **错误做法**：
```
检测技术栈 → 自动写入 CLAUDE.md
```

✅ **正确做法**：
```
检测技术栈 → 用户确认 → 写入 CLAUDE.md
```

**原因**：自动检测可能不准确，必须用户确认。

---

### 反模式2：硬编码路径

❌ **错误做法**：
```
创建目录：/Users/username/.claude/
```

✅ **正确做法**：
```
检测操作系统 → 确定路径格式 → 创建目录
```

**原因**：不同操作系统路径格式不同。

---

### 反模式3：不可恢复的初始化

❌ **错误做法**：
```
执行步骤1 → 执行步骤2 → 执行步骤3 → 完成
（中断后需要从头开始）
```

✅ **正确做法**：
```
执行步骤1 → 记录完成 → 执行步骤2 → 记录完成 → 执行步骤3 → 完成
（中断后可从上次完成的步骤继续）
```

**原因**：初始化可能被中断，需要支持断点续传。

---

### 反模式4：跳过用户确认

❌ **错误做法**：
```
检测到配置冲突 → 自动覆盖
```

✅ **正确做法**：
```
检测到配置冲突 → 询问用户 → 确认/取消
```

**原因**：自动覆盖可能导致数据丢失。
