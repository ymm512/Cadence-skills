## 关键设计决策

1. **依赖关系灵活性**
   - Requirement/Analyze 不是必须前置，支持已有文档
   - 可根据流程模式（完整/快速/探索）跳过

2. **审查报告闭环**
   - Design Review → 发现问题 → 返回 Design（带着审查报告）→ 修改 → Design Review
   - P0 问题必须解决，允许标记为技术债务

3. **职责边界清晰**
   - Requirement: 业务需求（功能、流程、验收标准）
   - Design: 技术方案（架构、数据模型、API）
   - Plan: 实现计划（任务分解、依赖关系、并行识别）
   - Design Review: 审查和反馈
   - Git Worktrees: 环境隔离（分支创建、路径管理、环境验证）
   - Subagent Development: 代码实现（TDD 流程、单元测试、代码审查）

4. **Plan 节点关键特性**
   - 任务粒度标准：1-4 小时，可独立完成
   - 并行任务识别：支持 Subagent 并发执行
   - 时间估计：基于复杂度给出时间范围
   - 不识别风险：风险在 Design Review 阶段完成

5. **Git Worktrees 节点关键特性**
   - Git 状态检查：主分支最新、工作区干净（强制）
   - 分支命名：基于功能名称自动生成（`feature/{feature-name}`）
   - 路径策略：独立目录（`../workspace/{feature-name}`）
   - 环境验证：自动验证 Git 状态，手动验证依赖安装
   - 不管理多个 worktree：简化为单 worktree 记录

6. **Subagent Development 节点关键特性**
   - TDD 强制执行：RED-GREEN-BLUE 三阶段验证
   - 代码审查：每个任务完成后自动审查
   - 覆盖率标准：P0 必须 ≥ 80%
   - 并行执行：支持最多 49 个并发（默认 5 个）
   - 失败处理：自动重试 + 人工介入
   - 不可跳过：所有开发任务必须通过此节点
