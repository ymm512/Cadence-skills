## 总体评价

整体方案分层合理、职责边界清晰，作为一期的 issue 级闭环设计已经比较完整。以下从架构、角色模型、协议与工件、命令面、缺失章节等维度给出具体评审意见。

### 值得肯定的设计点

1. **分层职责清晰** — Vibe Kanban / OpenSpec / superpowers / Aria 四层边界明确
2. **松耦合原则** — 不内嵌、不 fork 的约束很合理
3. **双入口设计** — 支持 VK 和 native 入口，灵活性好
4. **fast-lane 通道** — 避免所有任务都走重流程，实用
5. **扩展接口预留** — branch/PR lifecycle 的预留位设计合理

## 问题清单

### 一、架构层面

#### [P0] 1. Codex 触发机制未说明

方案多次提到"Claude Code 驱动 Codex"，但完全没有描述：

- Claude Code 通过什么机制触发 Codex？是 subagent、CLI 调用、还是 API 调用？
- 触发失败或超时的容错策略是什么？
- Codex 进程的生命周期由谁管理？

**影响**：这是整个方案的核心链路，缺少这部分内容会让实现者无法下手。

#### [P0] 2. "并行执行"缺乏具体描述

方案提到"Claude Code 能驱动 Codex 并行执行 exec"和"review 与 test 能独立并行"，但缺少：

- 并行的粒度是什么？多个 Codex 实例？一个 Codex 实例内的多任务？
- 并行的资源隔离策略（worktree 如何分配、文件冲突如何避免）
- 并行上限与背压策略

#### [P0] 3. 状态持久化方案缺失

状态机定义了 10 个状态，但没有说明：

- 状态存储在哪里？文件？内存？Git note？
- 任务中断后如何恢复？用户关闭 Claude Code 再打开，能从哪个状态继续？
- 多任务并行的状态隔离怎么做？

---

### 二、角色模型

#### [P1] 4. orchestrator 定义过于模糊

方案说 orchestrator 是"系统控制层"，但只给了一句话描述。缺少：

- orchestrator 本身是 skill、command 还是 runtime 逻辑？
- orchestrator 与 8 个业务角色的调用关系（谁调谁、何时调）
- orchestrator 是否有独立状态？还是无状态的调度器？

#### [P1] 5. fast-lane 准入标准不明确

方案给了 5 条升级规则（从 fast-lane 升回正式流），但缺少正向判定标准：

- 什么条件下一个任务**可以**进入 fast-lane？
- "小修小补"的定义太模糊，实现者会自行发挥
- fast-lane 的执行主体是谁？还是 Codex 吗？还是 Claude Code 直接执行？

**建议**：给出明确的准入 checklist，比如：单文件改动、不超过 N 行、无跨模块影响等。

---

### 三、协议与工件

#### [P0] 6. 运行时工件格式未定义

方案列出了 8 种运行时工件（task intake card、dispatch contract 等），但全部没有给出：

- 格式规范（JSON？YAML？Markdown？）
- 必填字段定义
- 工件之间的引用关系

**影响**：这些是 role 之间传递信息的契约，没有定义就无法实现。

#### [P1] 7. dispatch contract 和 patch contract 区分不清

"dispatch contract"是 Claude 给 Codex 的执行契约，"patch contract"是 review/test 失败后给 Codex 的修补契约。两者格式是否相同？patch contract 是否引用 dispatch contract？是否有继承关系？都没有说明。

#### [P1] 8. 质量门的定义缺失

方案多次提到"质量门"，但完全没有定义：

- 质量门包含哪些检查项？
- 谁定义质量门？（plan 角色？还是 Aria 默认？）
- 质量门是全局的还是按任务类型可配的？

---

### 四、命令面

#### [P1] 9. aria:start 与 aria:run 的衔接点不清晰

- `aria:start` 驱动 `intake -> spec -> plan`，结束后状态是 `planned`
- `aria:run` 从 `dispatched` 开始
- 但 `planned -> dispatched` 这个状态转换发生在哪里？是 `aria:run` 的第一步？还是需要用户手动确认？

方案说"用户确认计划后执行 aria:run"，但没有定义确认机制。

#### [P2] 10. aria:status 和 aria:result 缺少上下文

多任务场景下，`aria:status` 显示的是哪个任务？是当前活跃任务？所有任务？需要 task ID 参数吗？

---

### 五、缺失的重要章节

#### [P1] 11. 错误处理策略

整个方案没有任何关于异常情况的处理说明：

- Codex 执行崩溃怎么办？
- patch 循环无限进行怎么办？（建议设定最大 patch 轮次）
- OpenSpec 工件创建失败怎么办？
- 用户中途想取消任务怎么办？

#### [P1] 12. 依赖前置条件检查

方案说 OpenSpec 和 superpowers 是前置 plugin，但没有定义：

- 安装检查机制是什么？（探测文件？探测命令？）
- 版本兼容性如何声明？
- 缺失时的错误信息和降级策略

#### [P2] 13. 测试与验证策略

作为一期交付，方案缺少：

- 如何验证 Aria 自身的正确性？
- 有没有端到端测试流程？
- 有没有最简验收场景定义（比如一个最简 issue 的完整闭环 demo）？

#### [P2] 14. 与现有项目目录结构的关系

方案建议的目录 `cadence-aria/` 放在项目根目录，但当前项目的 `cadence/` 目录已有约定（文档存储规则）。需要明确：

- `cadence-aria/` 是否遵循 `cadence/` 文档存储规则？
- 两个目录的职责边界在哪？

## 建议优先补充的内容

| 优先级 | 缺失项 | 理由 |
|--------|--------|------|
| **P0** | Codex 触发机制 | 核心链路，无此无法实现 |
| **P0** | 状态持久化方案 | 决定任务能否跨会话存活 |
| **P0** | 运行时工件格式 | 角色间契约，实现必须 |
| **P1** | 错误处理策略 | 决定系统鲁棒性 |
| **P1** | fast-lane 准入标准 | 模糊会导致滥用或误判 |
| **P1** | 质量门定义 | 验证闭环的关键依据 |
| **P1** | orchestrator 细化 | 影响实现复杂度 |
| **P1** | 依赖检查机制 | 影响用户体验 |
| **P2** | 最简验收场景 | 验证一期是否成功 |
| **P2** | aria:status/result 多任务上下文 | 命令面完整性 |
| **P2** | 目录结构关系 | 项目组织一致性 |

## 评审结论

方案在架构分层、角色划分、流程设计上已经比较成熟，但存在 **3 个 P0 级关键缺失**（Codex 触发机制、状态持久化、工件格式），这些问题直接影响实现可行性。

建议在进入实现前，至少补充 P0 和 P1 级内容，形成 v1.1 版本后再启动开发。
