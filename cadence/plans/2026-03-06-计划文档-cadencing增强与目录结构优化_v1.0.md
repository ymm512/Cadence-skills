# 设计文档：Cadencing Skill 增强与目录结构优化

**文档信息**
- **设计编号**：DES-2026-03-06-001
- **创建日期**：2026-03-06
- **版本**：v1.0
- **负责人**：Claude
- **关联需求**：优化项目初始化流程，增强定制化能力

## 1. 设计概述

### 1.1 设计目标

1. **增强 cadencing skill**：添加项目分析功能和个性化规则支持
2. **优化目录结构**：统一和扩展 `.claude/` 目录结构
3. **支持定制化**：允许用户定义自己的文档模板和开发规范

### 1.2 设计范围

- cadencing skill 的功能增强
- 相关 skills 的输出路径调整
- CLAUDE.md 目录结构规范更新
- 项目个性化规则文档示例创建

### 1.3 设计原则

- **向后兼容**：不破坏现有功能
- **用户控制**：定制化由用户主动启用
- **示例引导**：提供清晰的示例文档
- **统一规范**：所有文档遵循统一的目录结构

## 3. 详细设计

### 3.1 cadencing skill 修改

#### 3.1.1 新增步骤 2.5：项目分析

**位置**：在步骤 2（Claude Code 初始化）之后

**分析内容**：

```markdown
1. 项目基本信息
   - 项目类型（前端/后端/全栈/其他）
   - 主要编程语言
   - 项目规模（文件数、目录数、代码行数）

2. 目录结构
   - 主要目录识别
   - 项目结构模式

3. 依赖关系
   - 依赖文件识别
   - 主要依赖列表

4. 模块组件
   - 主要模块/组件
   - 入口文件

5. Git 历史
   - 最近 10 条提交
   - 提交活跃度
```

**输出文件**：`.claude/analysis-docs/YYYY-MM-DD_分析报告_项目初始化摘要_v1.0.md`

**文件内容结构**：

```markdown
# 项目初始化分析摘要

**生成时间**：YYYY-MM-DD HH:MM:SS
**项目路径**：[路径]

## 1. 项目基本信息
- 项目类型：[类型]
- 主要语言：[语言]
- 项目规模：[统计]

## 2. 目录结构
[目录树 + 说明]

## 3. 依赖关系
[主要依赖列表]

## 4. 主要模块
[模块说明]

## 5. Git 历史
[提交记录 + 统计]

## 6. 下一步建议
[基于分析的建议]
```

#### 3.1.2 新增步骤 11：创建项目个性化规则文档案例

**位置**：在步骤 10（创建目录结构）之后

**创建内容**：

```
.claude/project-rules/
├── README.md                    # 使用说明
└── examples/                     # 示例目录
    ├── requirement-template.md  # 需求文档模板
    ├── design-template.md       # 设计文档模板
    ├── coding-standards.md      # 代码开发规范

**示例文件特点**：
- 包含详细注释和使用说明
- 提供完整的模板结构
- 展示最佳实践
- 不自动启用，仅作参考

**CLAUDE.md 中的示例规则**（注释状态）：

```markdown
## 项目个性化规则（示例）

> 以下是使用示例，默认不启用。
> 如果您创建了自定义规则，可以取消注释或添加类似规则：

<!--
### 需求文档格式
使用 `.claude/project-rules/requirement-template.md` 作为需求文档格式。

### 设计文档格式
使用 `.claude/project-rules/design-template.md` 作为设计文档模板。
-->
```

#### 3.4.2 添加个性化规则使用说明

新增章节：

```markdown
### 6. 项目个性化规则

> **🔴 定制化能力**

- **规则目录**：`.claude/project-rules/`
- **使用方法**：
  1. 查看项目初始化时创建的示例文件（`examples/` 目录）
  2. 根据需要复制和修改示例文件
  3. 在本文件（CLAUDE.md）中添加规则，指导 Claude 使用您的定制文档

**示例规则**：

```markdown
## 项目个性化规则

### 需求文档格式
使用 `.claude/project-rules/requirement-template.md` 作为需求文档格式，
不要使用 requirement skill 中的通用格式。

### 设计文档格式
使用 `.claude/project-rules/design-template.md` 作为设计文档模板。

### 代码开发规范
所有代码开发必须遵循 `.claude/project-rules/coding-standards.md` 中的规范。
```

**说明**：
- 个性化规则由用户主动启用
- Claude Code 会自动遵循 CLAUDE.md 中的规则
- 示例文件仅作参考，不强制使用
```

## 4. 实现影响

### 4.1 向后兼容性

**影响评估**：
- ✅ **新项目**：完全使用新的目录结构和功能
- ⚠️ **现有项目**：
  - 如果存在 `analysis/` 目录，需要手动重命名为 `analysis-docs/`
  - 如果存在 `modao/` 目录，需要手动重命名为 `modaos/`
  - 如果存在 `model/` 目录，需要手动重命名为 `models/`
  - 新增的目录（prds/、designs-reviews/、project-rules/、reports/）不影响现有功能

**迁移建议**：
- 提供迁移脚本自动重命名目录
- 在 CLAUDE.md 中说明目录重命名的原因

### 4.2 依赖关系

**内部依赖**：
- cadencing skill 依赖所有其他 skills 的路径配置
- 其他 skills 独立运行，不受 cadencing 影响

**外部依赖**：
- 无新增外部依赖
- 使用现有的文件系统和 Claude Code 能力

### 4.3 性能影响

**项目分析性能**：
- 文件扫描：< 5秒（< 1000 文件）
- 依赖分析：< 2秒
- Git 历史分析：< 1秒
- 总体影响：可忽略（仅初始化时执行一次）

**规则加载性能**：
- 无性能影响（CLAUDE.md 已加载）
- 示例文件不参与运行时逻辑

## 5. 风险评估

| 风险 | 影响 | 概率 | 应对措施 |
|------|------|------|---------|
| 目录重命名导致路径错误 | 高 | 中 | 提供迁移脚本，更新所有 skills 的路径配置 |
| 用户不理解个性化规则用法 | 中 | 高 | 提供详细的 README 和示例，引导用户正确使用 |
| 项目分析耗时过长 | 低 | 低 | 设置超时限制，显示进度提示 |
| 示例文件过多占用空间 | 低 | 低 | 示例文件精简，总大小 < 50KB |
