### 🆕 问题6：数据模型不一致且不完整

**🟡 严重程度：中等**

**问题描述**：

TodoWrite 和 Serena memory 的数据模型**不一致**，且都**不完整**，无法支持完整的进度追踪。

**详细证据**：

#### TodoWrite 的数据模型

根据 `subagent-development/SKILL.md`（lines 109-127）：

```json
{
  "tasks": [
    {
      "id": "1",
      "subject": "任务1标题",
      "description": "完整任务描述...",
      "status": "pending"
    }
  ]
}
```

**包含的信息**：
- ✅ 任务 ID
- ✅ 任务标题
- ✅ 任务描述
- ✅ 任务状态（pending/in_progress/completed）
- ✅ 依赖关系（blockedBy）

**缺少的信息**：
- ❌ 开始时间
- ❌ 结束时间
- ❌ 已耗时
- ❌ 进度百分比
- ❌ 重试次数
- ❌ 测试结果
- ❌ 审查结果
- ❌ 覆盖率
- ❌ Git commits

#### Serena memory 的数据模型

根据 `full-flow/SKILL.md`（lines 122-132）的示例：

```javascript
write_memory({
  memory_name: `checkpoint-full-flow-brainstorm-${timestamp}`,
  content: {
    phase: "brainstorm",
    status: "completed",
    output: "PRD文档路径",
    timestamp: new Date().toISOString()
  }
})
```

**包含的信息**：
- ✅ 阶段名称（phase）
- ✅ 状态（status）
- ✅ 输出产物（output）
- ✅ 时间戳（timestamp）

**缺少的信息**：
- ❌ 任务级别信息（Task ID、任务标题）
- ❌ 详细进度信息（百分比、耗时）
- ❌ 测试和审查结果
- ❌ Git 提交信息
- ❌ 统计信息

#### 数据模型不一致

| 信息类型 | TodoWrite | Serena memory | 一致性 |
|---------|-----------|--------------|--------|
| 任务 ID | ✅ 有 | ❌ 无 | ❌ 不一致 |
| 任务状态 | ✅ 有 | ✅ 有 | ✅ 一致 |
| 开始时间 | ❌ 无 | ❌ 无 | ⚠️ 都没有 |
| 结束时间 | ❌ 无 | ❌ 无 | ⚠️ 都没有 |
| 进度百分比 | ❌ 无 | ❌ 无 | ⚠️ 都没有 |
| 测试结果 | ❌ 无 | ❌ 无 | ⚠️ 都没有 |
| 阶段信息 | ❌ 无 | ✅ 有 | ❌ 不一致 |

**结论**：TodoWrite 和 Serena memory 的数据模型**不一致且不完整**，无法支持完整的进度追踪。

**结论**：**假设不成立**

## 3️⃣ 根本原因深度分析

### 核心问题：设计与实现完全脱节

**问题根源**：

```
设计文档 → 描述"应该做什么"
         ↓
         ❌ 缺少"如何做"
         ↓
实现代码 → 不存在
```

**具体表现**：

1. **Commands 层面**：
   - 有完整的用户文档（"应该输出什么"）
   - 没有执行逻辑（"如何获取数据"）
   - **结果**：无法执行

2. **Skills 层面**：
   - 流程 skills 有示例代码（"可以这样保存 checkpoint"）
   - 节点 skills 没有实现（"实际不会保存"）
   - **结果**：不会执行

3. **数据层面**：
   - 有数据结构描述（"应该存储什么"）
   - 没有数据模型定义（"如何存储和读取"）
   - **结果**：无法存储

4. **触发机制层面**：
   - 有触发时机描述（"应该在什么时候触发"）
   - 没有触发器实现（"如何监听事件"）
   - **结果**：无法触发

### 设计缺陷

#### 缺陷1：缺少"执行层"

```
用户输入 → /status → ??? → 输出结果
                    ↑
                    缺少这一层
```

**应该有的架构**：
```
用户 → Command (接口层) → Skill (执行层) → 工具层 (Serena/TodoWrite)
```

**实际的架构**：
```
用户 → Command (接口层) → ❌ 缺失 → 输出结果
```

---

#### 缺陷2：缺少"数据层"

```
进度数据 → ??? → Commands
          ↑
          缺少数据层
```

**应该有的数据流**：
```
节点执行 → 写入进度数据 → 存储层 → Commands 读取 → 输出
```

**实际的数据流**：
```
节点执行 → ❌ 不写入 → ❌ 没有存储 → Commands 无法读取
```

---

#### 缺陷3：缺少"事件层"

```
状态变更 → ??? → 自动触发
          ↑
          缺少事件层
```

**应该有的事件流**：
```
节点完成 → 触发事件 → 保存 Checkpoint
测试失败 → 触发事件 → 记录失败日志
会话中断 → 触发事件 → 保存 Session Summary
```

**实际的事件流**：
```
节点完成 → ❌ 没有事件 → ❌ 不保存
测试失败 → ❌ 没有事件 → ❌ 不记录
会话中断 → ❌ 没有事件 → ❌ 不保存
```

## 5️⃣ 解决方案分析

### 方案评估标准

1. **可行性**：技术上是否可实现
2. **完整性**：是否能实现所有功能
3. **维护性**：是否易于维护和扩展
4. **一致性**：是否符合现有架构

### 方案1：创建进度追踪 Skills（推荐，与 v1.0 一致）

**核心思路**：为每个 command 创建对应的 skill，实现执行逻辑。

#### 实施步骤（与 v1.0 一致，但更详细）

##### Step 1：定义数据模型

**统一数据模型**：

```yaml
Progress:
  # 项目信息
  project_info:
    name: string
    flow_type: "full-flow" | "quick-flow" | "exploration-flow"
    current_phase: string
    git_branch: string

  # 阶段进度
  phases:
    - phase_name: string
      status: "completed" | "in_progress" | "pending"
      start_time: timestamp | null
      end_time: timestamp | null
      tasks: Task[]

  # 整体进度
  overall_progress:
    percentage: number  # 0-100
    completed_phases: number
    total_phases: number

  # 时间统计
  time_stats:
    total_time: number  # 秒
    estimated_remaining: number  # 秒

Task:
  task_id: string
  subject: string
  description: string
  status: "pending" | "in_progress" | "completed" | "failed"
  start_time: timestamp | null
  end_time: timestamp | null
  retry_count: number
  test_result: "passed" | "failed" | null
  review_result: "passed" | "failed" | null
  coverage: number | null  # 0-100
  git_commits: string[]

Checkpoint:
  checkpoint_id: string
  timestamp: timestamp
  phase: string
  task_id: string | null
  status: string
  context: object
  output: string  # 文件路径
```

**数据存储位置**：
- **主存储**：Serena memory（持久化）
- **缓存层**：TodoWrite（临时状态）

**数据同步策略**：
- 写入：同时写入 Serena + TodoWrite
- 读取：优先从 Serena 读取，TodoWrite 作为辅助
