# 规则目录重构设计文档

> 日期：2026-04-09 | 版本：v1.1 | 状态：设计中

## 1. 背景与目标

### 现状问题

1. `CLAUDE.md` 约 620 行，7 大类规则全部内嵌，文件过大难以维护
2. 框架内置规则和用户自定义规则混在同一文件中，界限不清
3. `project-rules/` 目录定位不够明确，缺乏强制约束
4. 涉及规则创建的 commands/skills 缺乏统一的规则存储规范

### 目标

1. 新增 `.claude/rules/` 目录，存放框架内置规则
2. CLAUDE.md 只保留摘要 + 引用，详细规则拆分到 `rules/` 下的独立文件
3. 增加强制规则：用户自定义规则只能存放在 `project-rules/` 目录
4. 全面审查 commands/skills，确保规则创建遵循新规范

## 2. 目录结构设计

### 2.1 rules 目录（框架内置规则）

```
.claude/
├── rules/                          # 框架内置规则（维护者管理）
│   ├── README.md                   # 规则目录说明
│   ├── language.md                 # 语言规则
│   ├── code-usage.md               # 代码使用规则
│   ├── document-storage.md         # 文档存储规则（含命名规范、路径映射、禁止行为）
│   ├── markdown-format.md          # Markdown 格式规则
│   ├── serena-usage.md             # Serena MCP 使用规则
│   ├── mcp-servers.md              # MCP Server 使用规则（所有 MCP 工具）
│   └── playwright.md               # Playwright CLI 使用规则
```

### 2.2 project-rules 目录（用户自定义规则）

```
.claude/
├── project-rules/                  # 用户自定义规则（用户管理）
│   ├── README.md                   # 用户规则说明（含强制约束）
│   └── examples/                   # 规则模板示例
│       ├── coding-standards.md
│       ├── design-template.md
│       ├── requirement-template.md
│       └── test-standards.md
```

### 2.3 两个目录的职责边界

| 维度 | `rules/` | `project-rules/` |
|------|----------|-------------------|
| 管理者 | 框架维护者 | 用户 |
| 内容 | 通用规则（语言、文档、MCP 等） | 项目个性化规则（编码规范、模板等） |
| 修改权限 | 仅维护者 | 用户自由修改 |
| 创建方式 | 安装脚本自动创建 | 用户按需创建或从 examples 复制 |

## 3. CLAUDE.md 重构设计

重构后的 CLAUDE.md 精简为 ~100-120 行，分为"强制规则"和"项目配置"两大区域，结构如下：

````markdown
# CLAUDE.md

本文件为 Claude Code (claude.ai/code) 在此仓库中工作提供指导。

## 强制规则

> **🔴 必须遵守 - 无例外**
> 详细规则见 `.claude/rules/` 目录下的各规则文件。
> 用户自定义规则见 `.claude/project-rules/` 目录。

### 1. 语言规则
- **必须使用中文回答** → 详见 `.claude/rules/language.md`

### 2. 代码使用规则
- **非必要不编写代码**（Skills 项目特殊规定） → 详见 `.claude/rules/code-usage.md`

### 3. 文档存储规则
- **所有文档必须存放在 `.claude` 目录下** → 详见 `.claude/rules/document-storage.md`

### 4. Markdown 格式规则
- **代码块嵌套使用 4 反引号/3 反引号** → 详见 `.claude/rules/markdown-format.md`

### 5. Serena 使用规则
- **禁止分析 .git 目录** → 详见 `.claude/rules/serena-usage.md`

### 6. MCP Server 使用规则
- **各 MCP 工具的使用规范** → 详见 `.claude/rules/mcp-servers.md`

### 7. 项目个性化规则（强制规则）
- **用户自定义规则只能存放在 `.claude/project-rules/` 目录**
- 禁止在 `rules/` 目录中添加用户自定义规则
- 禁止直接修改 `rules/` 目录下的框架内置规则文件
- 详见 `.claude/project-rules/README.md`

### 8. Playwright CLI 使用规则
- **浏览器自动化工具规范** → 详见 `.claude/rules/playwright.md`

## 项目配置

> 以下内容由初始化脚本根据项目环境自动检测生成，非通用规则。

### 包管理器规则
- **前端项目**：必须使用 `pnpm` 作为包管理器
- **Python 项目**：必须使用 `uv` 作为包管理器
- **禁止使用**：npm（前端）、pip（Python）、yarn（前端）

### 项目技术栈
- **语言**：[语言列表]
- **包管理器**：[pnpm/uv]
- **测试命令**：[命令]
- **检查命令**：[命令]
- **格式化命令**：[命令]
- **覆盖率阈值**：80%

## 项目信息
# currentDate
Today's date is {当前日期}。
````

> **设计说明**：将"包管理器规则"和"技术栈检测"保留在 CLAUDE.md 中作为"项目配置"区域，因为它们是初始化时动态生成的项目特定信息，不属于通用规则，不适合放入 `rules/` 或 `project-rules/`。

### 关键设计决策

1. 每个规则只保留 **1 行摘要 + 文件引用路径**
2. 新增 **第 7 条强制规则**，明确 `project-rules/` 是用户自定义规则的唯一合法位置
3. 明确禁止用户修改 `rules/` 目录
4. **"包管理器规则"和"技术栈检测"保留在 CLAUDE.md** 中作为"项目配置"区域，因为它们是初始化时动态检测的项目特定信息，不属于通用规则

### 极短规则文件的保留理由

`language.md`（约 3 行）和 `markdown-format.md`（约 10 行）虽然内容极少，但保留为独立文件的理由：
- 保持每个规则域的独立性，职责单一
- 未来可能扩展（如 language.md 可能增加多语言项目规则）
- 用户通过 `project-rules/` 覆盖时，引用路径更清晰

### 5.2 新增 references/rules/ 模板目录

为了集中管理框架内置规则的内容模板，新增 `cadence-init/references/rules/` 目录：

```
cadence-init/references/rules/
├── README.md                   # rules 目录说明模板
├── language.md                 # 语言规则模板
├── code-usage.md               # 代码使用规则模板
├── document-storage.md         # 文档存储规则模板
├── markdown-format.md          # Markdown 格式规则模板
├── serena-usage.md             # Serena 使用规则模板
├── mcp-servers.md              # MCP Server 使用规则模板
└── playwright.md               # Playwright CLI 使用规则模板
```

**设计理由**：

- 将规则内容集中在 `references/rules/` 中管理，避免硬编码在 command 文件中
- `rule-config.md` 和 `mcp-configuration.md` 执行时从该目录读取模板，写入用户项目的 `.claude/rules/`
- 与现有的 `references/project-rules/` 结构对称

### 5.3 需要审查的文件

以下文件需要检查是否有硬编码的规则路径引用或规则创建逻辑：

| 文件 | 审查重点 |
|------|---------|
| `cadence-init/commands/pre-check.md` | 是否检查规则路径 |
| `cadence-init/commands/project-analysis.md` | 是否引用规则路径 |
| `cadence-init/commands/skill-create.md` | skill 创建是否涉及规则路径 |
| `cadence-workflow/commands/*.md` | 是否有硬编码规则路径 |

### 5.4 审查标准

对所有 commands/skills 的审查遵循以下标准：

1. **规则存储**：框架内置规则 → `rules/`，用户自定义 → `project-rules/`
2. **CLAUDE.md 引用**：只写摘要 + 引用，不写规则原文
3. **强制约束**：明确告知用户 `project-rules/` 是唯一合法的自定义规则目录

### 5.5 已有项目迁移方案

对于已使用旧版 cadence-init 初始化的项目：

1. **手动迁移**：用户重新运行 `/rule-config` 命令，命令将自动检测已有 CLAUDE.md 内容并转换为新的摘要+引用格式
2. **不自动迁移**：为避免误操作，不提供自动迁移脚本
3. **迁移指导**：在 `rules/README.md` 中增加"从旧版迁移"章节，说明手动迁移步骤

### 5.6 实施顺序

1. 创建 `cadence-init/references/rules/` 目录和模板文件
2. 修改 `cadence-init/commands/rule-config.md`
3. 修改 `cadence-init/commands/mcp-configuration.md`
4. 更新 `cadence-init/references/project-rules/CLAUDE-RULE.md`
5. 更新 `cadence-init/references/project-rules/README.md`
6. 修改 `cadence-init/commands/project-rules-examples.md`
7. 重构 `CLAUDE.md`（本项目）
8. 审查其他 commands/skills

## 6. 不在本次范围内

以下内容不在本次设计范围内，作为后续任务：

## 7. 验收标准

1. `CLAUDE.md` 精简至 ~120 行以内，只含摘要+引用和项目配置
2. `.claude/rules/` 目录下有 7 个规则文件 + 1 个 README
3. 逐行对比拆分后文件内容与原 CLAUDE.md 对应行号范围，确保无遗漏
4. `cadence-init/references/rules/` 目录有对应的 7+1 个模板文件
5. `project-rules/` 的 README 和模板更新，含强制约束说明和完整使用指导
6. `rule-config.md`、`mcp-configuration.md`、`project-rules-examples.md` 已更新
7. 所有相关 commands/skills 已审查并按需修改
