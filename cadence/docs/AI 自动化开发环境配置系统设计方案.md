# AI 自动化开发环境配置系统设计方案

> 设计日期: 2026-02-04
> 设计目标: 构建一系列开发所需的 Agent，实现从需求到测试的全流程自动化

---

## 1. 系统概述

### 1.1 背景

作为研发负责人，需要构建一套 AI 自动化开发环境，覆盖需求整理、方案设计、代码生成、单元测试、业务测试等关键环节，提升团队开发效率。

### 1.2 核心特点

- **混合模式的多 Agent 协作系统**：流水线执行 + 人工跳转
- **状态驱动**：每个 Agent 的输出作为状态存储，支持断点续传
- **关键节点确认**：仅在设计评审、代码审查、合并请求等节点暂停
- **上下文继承**：Agent 之间通过标准化的上下文对象传递信息

### 1.3 技术栈

- 全栈开发（前后端均有）
- 大型存量项目（代码库庞大）
- Markdown 格式需求文档

## 3. Agent 详细设计

## 3.1 需求整理 Agent

### 3.1.1 输入

- PRD 文档（Markdown）
- 业务规则文档（Markdown）
- 项目背景知识（可选）

### 3.1.2 处理流程

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  读取 PRD    │ → │ 提取业务规则 │ → │  模块划分    │ → │  输出结构化  │
```

```json
{
```

## 3.2 方案设计 Agent

### 3.2.1 输入

- 需求整理 Agent 的输出（结构化需求）
- 设计规范文档（开发模式、目录划分、数据模型、组件调用关系）
- 实现思路（人工输入）
- 存量代码分析结果（如果是存量业务）

### 3.2.2 处理流程

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  判断业务    │ → │  分析存量    │ → │  生成技术    │ → │  输出设计    │
│  类型        │    │  代码/依赖   │    │  方案        │    │  文档        │
│(新/存量)     │    │  (如需要)    │    │              │    │              │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
        │                  │                  │
        ▼                  ▼                  ▼
   ┌─────────┐       ┌─────────┐       ┌─────────────┐
   │ 新功能:  │       │存量功能:│       │ 调用 LLM    │
   │ 全新设计 │       │ 代码分析│       │ 生成方案    │
   └─────────┘       └─────────┘       └─────────────┘
```

### 3.2.3 业务类型判断逻辑

| 判断条件 | 处理方式 |
|---------|---------|
| 模块类型为 `new` | 按全新功能设计 |
| 模块类型为 `existing` | 分析存量代码，设计增量修改方案 |
| 涉及多个存量模块 | 分析模块间依赖，设计集成方案 |

### 3.2.4 存量代码分析（关键能力）

```
┌─────────────────────────────────────────────────────────────┐
│                    存量代码分析引擎                          │
├─────────────────────────────────────────────────────────────┤
│  1. 代码检索：根据需求关键词定位相关文件                      │
│  2. 依赖分析：分析模块间的调用关系                            │
│  3. 接口识别：提取现有 API 接口定义                           │
│  4. 数据模型：识别数据库表结构和实体关系                      │
│  5. 影响评估：评估修改对现有功能的影响范围                    │
└─────────────────────────────────────────────────────────────┘
```

### 3.2.6 提示词配置

| 配置项 | 说明 |
|--------|------|
| `NEW_FEATURE_DESIGN_PROMPT` | 新功能设计的提示词模板 |
| `EXISTING_FEATURE_DESIGN_PROMPT` | 存量功能改造的提示词模板 |
| `CODE_ANALYSIS_PROMPT` | 存量代码分析的提示词模板 |
| `ARCHITECTURE_DESIGN_PROMPT` | 架构设计的提示词模板 |
| `API_DESIGN_PROMPT` | API 设计的提示词模板 |
| `DATA_MODEL_PROMPT` | 数据模型设计的提示词模板 |

### 3.2.7 人工补充内容

- 实现思路描述（必须）
- 设计规范文档路径
- 存量代码分析范围（如果是存量业务）

### 3.2.8 人工确认环节

- **设计方案评审**：确认技术方案可行性，包括：
  - 架构设计是否合理
  - 数据模型变更是否正确
  - 与存量系统的集成方式
  - 文件变更列表是否完整

## 3.3 代码生成及单元测试 Agent

### 3.3.1 输入

- 方案设计 Agent 的输出（设计文档）
- 代码规范配置
- 项目模板/脚手架

### 3.3.2 处理流程

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  创建 Git    │ → │  生成代码    │ → │  生成单元    │ → │  执行单元    │
│  分支        │    │  (人工确认)  │    │  测试        │    │  测试        │
│  (人工触发)  │    │              │    │  (人工确认)  │    │              │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
                                                          │
                              ┌───────────────────────────┘
                              ▼
                    ┌─────────────────┐
                    │  测试是否通过?   │
                    └─────────────────┘
                          │         │
                    通过  ▼         ▼ 失败
                    ┌─────────┐  ┌─────────────┐
                    │ 代码提交 │  │ 调试修复     │
                    │ (人工触发)│  │ (自动/人工)  │
                    └─────────┘  └─────────────┘
                          │
                          ▼
                    ┌─────────────┐
                    │ 发起 Merge   │
                    │ Request      │
                    │ (人工触发)   │
                    └─────────────┘
```

### 3.3.3 代码生成模块

```
┌─────────────────────────────────────────────────────────────┐
│                    代码生成引擎                              │
├─────────────────────────────────────────────────────────────┤
│  1. 模板选择：根据设计文档选择对应代码模板                    │
│  2. 上下文注入：将设计信息注入模板                            │
│  3. 代码生成：调用 LLM 生成具体代码                           │
│  4. 规范检查：检查代码是否符合项目规范                        │
│  5. 依赖注入：自动添加必要的 import/依赖                      │
└─────────────────────────────────────────────────────────────┘
```

### 3.3.4 单元测试生成模块

```
┌─────────────────────────────────────────────────────────────┐
│                    单元测试生成引擎                          │
├─────────────────────────────────────────────────────────────┤
│  1. 测试策略：根据代码类型选择测试策略                        │
│     - Controller → 集成测试 (API 调用)                       │
│     - Service → 单元测试 (Mock 依赖)                         │
│     - Repository → 数据访问测试                              │
│     - Component → 组件渲染测试                               │
│  2. 用例生成：基于业务规则生成测试用例                        │
│  3. 边界值分析：自动生成边界条件测试                          │
│  4. Mock 生成：自动生成依赖的 Mock                            │
│  5. 测试数据：生成测试所需的 Fixture 数据                     │
└─────────────────────────────────────────────────────────────┘
```

### 3.3.5 调试修复模块（自动/人工）

## 5. 实施建议

### 5.1 阶段划分

| 阶段 | 目标 | 周期 |
|-----|------|------|
| 第一阶段 | 实现需求整理 Agent + 方案设计 Agent | 2-3 周 |
| 第二阶段 | 实现代码生成 Agent（新功能场景） | 2-3 周 |
| 第三阶段 | 实现存量代码分析 + 单元测试生成 | 2-3 周 |
| 第四阶段 | 实现业务测试 Agent + 全流程打通 | 2 周 |

### 5.2 风险提示

1. **存量代码分析复杂度**：大型项目的代码理解需要强大的 RAG 能力
2. **代码生成质量**：需要建立完善的代码审查机制
3. **安全合规**：Git 操作需要严格的权限控制
